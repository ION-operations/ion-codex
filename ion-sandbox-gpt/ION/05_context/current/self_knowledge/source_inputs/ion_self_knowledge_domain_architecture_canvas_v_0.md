# ION Self-Knowledge Domain Architecture Canvas v0.2

## Status

This document is a working canvas for activating ION's self-knowledge architecture. It is not accepted state, not final canon, and not a replacement for existing ION law. It exists to prevent the repeated failure where a new carrier tries to reconstruct ION from scattered files instead of mounting a routed ION-about-ION domain.

## Core correction

ION is not missing the idea of domains, agents, context systems, templates, route maps, or self-description. Those organs already exist across the current GPT package, full local root, registries, protocols, and product materials.

The failure is not absence.

The failure is that ION's knowledge of itself is not yet activated as a clean, first-class, routable self-knowledge domain that every carrier must mount before interpreting ION.

The mission is therefore:

```text
recover existing self-knowledge organs
→ classify and rank them
→ route them into domains
→ build self-knowledge context packages
→ mount future carriers through those routes
→ prevent repeated foundational misunderstanding
```

Do not describe this as a new architecture. It is an activation and organization of ION's existing architecture.

## Non-negotiable distinction

ION has one law and many host bodies.

```text
Full local/API-style ION
= intended full multi-invocation body, using isolated agent calls.

Codex CLI local ION
= practical local/API-like body, able to run separate role invocations.

Custom GPT / ChatGPT sandbox ION
= full ION adapted to sandbox constraints, with single-carrier fallback only when the host cannot create isolated invocations.
```

Single-carrier sequential mode is not the conceptual base of ION. It is a host-limited compatibility mode.

The canonical workflow remains ION's workflow. The carrier adapter changes the execution body.

## Why the self-knowledge domain is necessary

Without a mounted self-knowledge system, carriers are forced to infer ION from:

- architecture documents,
- package manifests,
- active work packets,
- historical donor archives,
- journals,
- tests,
- local runtime code,
- product instructions,
- stale recovery residue,
- and Braden's corrections.

This creates the recurring failure:

```text
carrier reads fragments
→ carrier guesses ION's shape
→ carrier promotes the wrong surface
→ carrier invents a replacement or reduction
→ Braden has to restate the obvious
→ the new artifact becomes more residue
```

ION should not require archaeological reconstruction before every meaningful action.

Future carriers should mount:

```text
ION_SELF_KNOWLEDGE_SYSTEM
→ query classification
→ domain route
→ depth ladder
→ authority-ranked evidence
→ anti-drift notes
→ bounded answer or action
```

## Self-knowledge specialist roles

These are not invented personalities. They are existing or implied ION specialist roles whose subject matter is ION itself.

### IONOLOGIST

Explains ION accurately at the requested depth. Owns identity and conceptual clarity.

### CANON_LIBRARIAN

Ranks sources, identifies active law, separates witness/donor/stale surfaces.

### CONTEXT_CARTOGRAPHER

Finds routes through ION knowledge, builds self-knowledge context packages.

### RUNTIME_CARTOGRAPHER

Explains embodiments: API, Codex CLI, GPT sandbox, MCP/Gateway, local runtime.

### TEMPLATE_CURATOR

Owns template law, template lineage, bindings, template gaps, and template anti-drift.

### ATLAS

Owns graph/domain structure, route maps, node relationships, and knowledge topology.

### VESTIGE

Handles historical/donor material without letting it become active law.

### NEMESIS

Audits contradictions, overclaims, stale resurfacing, and false reductions.

### STEWARD

Settles accepted self-knowledge changes and prevents fluent synthesis from becoming state without proof.

Example:

```yaml

  - full local ION

  - API style ION

  - ion_full_multi_agent
  - ion_carrier_embodiment
  - ion_agent_role
  - ion_context_system

  - single_carrier_is_fallback_not_base
```

## Host behavior

### API / Codex CLI / local full body

Use true isolated invocations when possible:

```text
compile role ContextPackage
→ invoke separate model/process/call
→ validate return
→ settle
→ receipt
```

### GPT sandbox body

Use fallback emulation:

```text
internal role sequence in one conversation
→ explicit warning of degraded isolation
→ preserve same proof/receipt law
```

The workflow law stays the same.

## Anti-drift rules that must be encoded immediately

1. Do not treat single-carrier mode as ION's intended center.
2. Do not treat Codex CLI full local ION as a helper for the GPT package.
3. Do not call GPT package “reduced ION.” It is full ION adapted to sandbox constraints.
4. Do not invent new ION law before checking existing domains, registries, templates, and protocols.
5. Do not let historical donor material become active authority.
6. Do not treat a passing narrow test as proof of full system readiness.
7. Do not turn broad synthesis into state without Steward/receipt.
8. Do not onboard carriers by dumping the whole repo.
9. Do not let fluent explanation replace route-based evidence.
10. Do not erase mature ION organs during “cleanup.”

A first accepted implementation should prove:

1. self-knowledge domains exist and parse,
2. each domain declares state class and authority posture,
3. carrier can mount a self-knowledge packet,
4. full multi-agent/API ION route exists and blocks single-carrier reduction,
5. GPT sandbox route exists and blocks product reduction,
6. stale/donor domain exists and prevents false authority,
7. at least one Codex/local workpacket uses the self-knowledge route,
8. package audit confirms self-knowledge included in GPT package,
9. tests pass,
10. no source is silently promoted to accepted state without Steward/receipt.
