# Proposed Browser Participation Posture For Phase 1

## Purpose

Settle the current-phase next step after the first disagreement drill without pretending
final staffing canon has been resolved.

## Proposal

- browser ChatGPT remains `EXTERNAL_UNMOUNTED` by default in Phase 1
- browser work must be governed by one `TASK`, `ROLE_SESSION`, or `HANDOFF`
- browser work must name explicit allowed targets and a bound template set
- browser returns must re-enter as `EXTERNAL_RETURN` plus `PATCH_PACKAGE` when edits are
  proposed
- live-branch truth and landing remain with the current live branch owner

## What changes operationally

- the branch now has one explicit disagreement drill demonstrating that mount disputes
  become visible artifact flows
- future browser participation is routed through the external zip-return bridge rather
  than direct role assignment
- the next proof target becomes Gate 5 browser zip return, not renewed mount debate

## Risks / edge cases

- browser ChatGPT may eventually deserve a stronger mounted posture, but that would need
  its own lawful mount packet and stronger current-phase evidence
- keeping browser external by default adds one extra landing step, but that is the price
  of preserving authority and provenance

## Open questions

- what exact packet family should govern the first real browser zip-return drill
- whether a future named browser role should exist at all, or whether external-unmounted
  plus bounded return is the better long-term posture
