# Canonicalization Decision: AIM-ION / AIM-OS Classification

## Purpose

Close q002 far enough that ION can stop speaking about AIM-ION / AIM-OS as a
half-remembered adjacent system and instead operate from an explicit
classification.

This packet does not decide any code import. It decides what AIM-ION on this
disk is, what AIM-OS proper is in relation to it, and how ION should treat both
within the current reintegration era.

## Scope and competing candidates

Objects in scope:

1. `AIM-ION/` on this disk
2. AIM-OS proper as referenced by `AIM-ION/` self-description

Classification options considered:

1. witness-only archive family
2. adjacent sibling universe / selective pattern reservoir
3. partial-adoption lane requiring near-term dedicated reintegration
4. dedicated reintegration lane or replacement-canon candidate

This decision is about classification, not adoption execution.

## Decision

### 1. Distinguish the two objects explicitly

`AIM-ION/` on this disk is not identical to AIM-OS proper.

The strongest local self-description is in `AIM_ION_OVERVIEW.md`, which names
`AIM-ION` as a curated copy / filesystem mirror of AIM-OS-GIT with regenerable
artifacts removed and Git history reset into a new orphan repository.

The root `README.md`, `AIM_OS_NORTH_STAR.md`, `SOURCE_OF_TRUTH.yaml`, and
`PROJECT_TRUTH/` pack then preserve AIM-OS-scale product, runtime, MCP, and
operator claims inside that curated mirror.

So the correct first distinction is:

- AIM-OS proper = referenced upstream organism and product claim surface
- `AIM-ION/` = local curated witness pack / mirror of that organism

### 2. Classification result

For current ION reintegration law, `AIM-ION/` is classified as an
**adjacent sibling witness pack and selective pattern reservoir**.

That means:

- it is not active ION canon,
- it is not a replacement-canon candidate,
- it is not currently a dedicated reintegration lane,
- it is not irrelevant historical debris,
- and it is permitted to guide later bounded extraction packets at the pattern
  level or the narrowly scoped interface level.

### 3. What is directly admissible as pattern guidance

The following are admissible as **selective pattern sources** for later ION
packets:

- operational gate writing and pass/fail boundary discipline from
  `PROJECT_TRUTH/05_operational_definition.md`
- smallest-real-path and operational-spine writing style from
  `PROJECT_TRUTH/06_operational_spine.md`
- compact machine-readable truth-pack generation pattern from
  `SOURCE_OF_TRUTH.yaml`
- bounded external context-package pattern from `context/README.md`
- provenance, witness, and model/provider explicitness as a design direction,
  not as a direct code import obligation

### 4. What is guidance only, not direct import

The following are useful to study but should not be imported directly in the
current era:

- broad memory stack and MCP substrate surfaces
- APOE / routing / orchestration ecosystem as a whole
- JOC / BAS / browser-centered operator runtime stack
- the larger AIM-OS product architecture and self-improving operating-system
  framing

These surfaces are too coupled to AIM-OS operational assumptions to be treated
as near-term ION imports.

### 5. What is explicitly out of scope for current ION reintegration

The following are explicitly out of scope for direct adoption in the current
reintegration cycle:

- consciousness / AGI vocabulary as an ION doctrinal import
- wholesale adoption of the AIM-OS subsystem stack
- wholesale adoption of the JOC / BAS / browser operating center
- using AIM-ION as a silent better canon that replaces ION’s current line

### 6. Why this is not a dedicated reintegration lane

The evidence does not support opening a dedicated AIM reintegration lane now.

Reasons:

- AIM-ION’s own surfaces are layered and partially contradictory
- `goals/STATUS.md` presents stale or temporally incoherent completion claims
- the strongest integration hooks to ION are historical, workspace, or mission
  references rather than a stable live contract
- the most powerful AIM surfaces are coupled stacks, not clean drop-in modules

So the right present classification is sibling-plus-pattern-reservoir, not
lane-first reintegration.

## Confidence and unresolved contradictions

Confidence: **high** that `AIM-ION/` is a curated local mirror/witness object
and should not be conflated with AIM-OS proper.

Confidence: **high** that the current classification should be adjacent sibling
plus selective pattern reservoir.

Confidence: **medium** on where the future extraction boundary should sit
between pattern-only borrowing and later interface-level reuse.

Unresolved contradictions that must remain visible:

- `AIM_ION_OVERVIEW.md` presents curation/orphan-mirror framing while the root
  `README.md` and `AIM_OS_NORTH_STAR.md` preserve AIM-OS-scale product language
- `goals/STATUS.md` carries stale or temporally incoherent completion language
  relative to other local files
- `SOURCE_OF_TRUTH.yaml` is machine-generated and useful, but its claims were
  not rerun inside this packet
- historical references to ION integration exist, but a stable direct
  AIM-to-ION contract was not found in this pass
