# Current-phase staffing and semantic identity archaeology (2026-04-12)

## Questions (from open thread)

1. Do older estates already distinguish **boot-carried** vs **semantic-record-carried** role identity?
2. Do they distinguish **support-field staffing** vs **constitutional rank classes**?
3. Do they distinguish **internal mounted roles** vs **external bounded carriers**?

## Active-branch findings (cited)

### 1. Boot-carried vs semantic-record — explicit fallback ladder

`ION/02_architecture/ROLE_CHASSIS_MOUNT_PROTOCOL.md` states that if a stronger semantic or registry identity exists, use it; if none exists yet, “the active boot plus protocol and packet assignment may carry the current-phase mount” (Core law items 4–5). Non-goals explicitly deny “every live role already has a full semantic identity record” (Non-goals §).

`ION/02_architecture/TRUE_NAME_AND_SEMANTIC_LAYER_PROTOCOL.md` §6 assigns governance precedence to `ION/03_registry/semantic_identities/*.semantic.yaml` when prose conflicts with a record — but §7 “Non-claims” admits no full dictionary, not all deep names, no final constitutional doctrine for every semantic class.

`ION/03_registry/semantic_identities/README.md` lists first-pass records only for Codex, Vizier, Vice, Nemesis — matching the research plan’s gap list for Mason, Thoth, Vestige, and browser.

**Conclusion for (1):** The live branch **already encodes** a settled *pattern*: semantic YAML when present **outranks** casual prose; otherwise **boot + mount packet + protocols** lawfully carry the role. That is **not** final global closure; it is a **two-tier recovery pattern** the branch chose deliberately.

### 2. Support staffing vs constitutional rank classes — parallel, not merged

`ION/02_architecture/RANK_AND_PRECEDENCE_PROTOCOL.md` maps four entities to rank classes (Vizier, Codex, Vice, Nemesis) and states it is “not a final constitutional hierarchy of all possible entities” and “does not yet define personnel law for every historical or future role” (Status, §5 scope).

`ROLE_CHASSIS_MOUNT_PROTOCOL.md` gives **current default interpretations** for Mason, Vestige, and browser (`EXTERNAL_UNMOUNTED`) without placing them in `RANK_AND_PRECEDENCE_PROTOCOL.md` rank classes — so **operational staffing defaults** and **first-pass rank taxonomy** are **intentionally decoupled** documents.

**Older estate — ION-BUILD:** `ION-BUILD/agents/` enumerates chassis-oriented trees (`CODEX`, `COMPOSER`, `FORGE`, `OPUS`, `SENTINEL`, `SEV`) — a **directory-native identity** pattern, not `*.semantic.yaml` in a single registry folder. `AETHER_INTERFACE.md` (constitution stratum) defines **execution permission classes** (observe → modify_policy, etc.) — a **capability lattice**, not the same four “rank classes” as ION’s migration-first-pass map.

**Older estate — ProjectOpus:** archaeology remap corpus references governance and boundaries (e.g. Aether kernel governance crosswalks); it does **not** provide a drop-in duplicate of ION’s 2026-04 mount + semantic_identities split (no single file was found that mirrors `ROLE_CHASSIS_MOUNT` + `semantic_identities/` together under those names).

**Conclusion for (2):** Older surfaces offer **different** decompositions (per-agent trees, execution classes, witness corpora). The active branch is **not** recovering one older file that already merged “support staffing law” with “constitutional rank”; it is **composing** a new explicit split: rank protocol partial, mount protocol for defaults, registry semantic subset partial.

### 3. Internal mounted vs external bounded — settled default in active law

`ROLE_CHASSIS_MOUNT_PROTOCOL.md` defines `EXTERNAL_UNMOUNTED` for browser ChatGPT by default and allows bounded external work without silent promotion to a named internal role (Mount postures §, browser subsection).

**Older estate:** ION-BUILD’s agent/composer split implies multiple **internal** chassis types; external browser governance in the **same** shape as ION’s `EXTERNAL_UNMOUNTED` + external-return bridge was **not** located in a quick path search (no requirement that it absent everywhere in Opus/SOS; only that **no homologous law file** surfaced under sibling roots for this pass).

**Conclusion for (3):** The **clearest** precedent for “external is explicitly not mounted until law says otherwise” is **current-branch** `ROLE_CHASSIS_MOUNT_PROTOCOL.md`, not a recovered Opus/SOS twin.

## Recommendation (archaeology-only)

The active root is **still in provisional bridge posture** for staffing/semantic closure **by its own law** (`PROVISIONAL_BRIDGE` / `NOT_FINAL_CANON` on mount protocol; explicit non-goals; partial semantic registry). Older estates supply **analogues** (agent directories, execution classes, Aether packets) but **not** a single pre-existing merged pattern identical to boot+ladder+partial semantic files+EXTERNAL_UNMOUNTED.

**For Codex consolidation:** Treat the next decision package as **evidence-first completion** of the deliberate two-tier pattern (who stays boot+mount-only, who gets `.semantic.yaml`, browser default unchanged unless explicit mount law), not as recovery of one lost “constitutional staffing” file from ION-BUILD/Opus.
