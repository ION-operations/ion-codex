# ION CUSTOM GPT KNOWLEDGE PACK LAYOUT

## Purpose

This document defines the initial curation layout for the ION custom GPT Knowledge pack.

The GPT Knowledge pack should be:
- compact
- canonical
- stable
- strongly relevant to every session

It should not become a dumping ground for mutable project state.

## Knowledge pack goal

A good Knowledge pack should let the GPT answer:

- what am I?
- what continuity artifact do I expect?
- how do I resume?
- how do I export?
- how do I preserve anti-drift continuity?
- what naming/packing conventions do I obey?

## Recommended modules

### Module A — Core shell law
- compact shell identity
- compact ION operating law for GPT mode
- anti-drift rules
- continuity-preservation rules

### Module B — Bundle doctrine
- working continuity bundle spec
- export protocol
- resume protocol
- compaction rules

### Module C — Stable canonical protocols
Only the minimal canon that the GPT must reliably operate from every session.

Recommended inclusions:
- continuity architecture
- temporal object schema
- orchestration temporal relevance
- temporal context lease
- triple-time reconciliation

### Module D — Artifact conventions
- deterministic naming conventions
- manifest rules
- continuity generation rules
- example bundle notes

### Module E — One or two compact examples
Only strong, minimal examples. Not giant archives.

## Knowledge pack selection rule

Prefer:
- stable
- compact
- always-relevant

Avoid:
- mutable
- noisy
- session-specific
- bulky

## Suggested first-pack file list

1. compact shell law
2. continuity bundle spec
3. working bundle schema
4. export protocol
5. resume protocol
6. continuity architecture
7. temporal object schema
8. orchestration temporal relevance
9. temporal context lease
10. triple-time reconciliation

This is intentionally conservative and should fit the first serious shell well.

## Present conclusion

The Knowledge pack should be the GPT’s stable doctrine mount, not its mutable continuity archive.
