# Proposed Decision Gates, Open Questions, and First Execution Order

## Purpose

This note compresses the foundry suite into the first actionable governance
sequence. Its aim is to prevent the new proposal surfaces from becoming another
beautiful but non-operative layer.

## Proposal

No new broad implementation should begin before the project resolves a small set
of explicit decisions in order.

The order matters because each decision constrains the next one.

## Decision gate 1: name the operation

The first decision is naming discipline.

My recommendation is:

- umbrella operation: **Constitutional Reintegration**
- live era: **Controlled Reintegration**
- current work class: **Lineage Adjudication**
- mechanism/workbench: **Canon Foundry**

This is not just branding. Each name refers to a different layer of truth, and
keeping those layers distinct helps avoid future confusion.

## Decision gate 2: choose the workspace authority strategy

The second decision is the root-authority strategy.

Three realistic options exist.

The first option is to promote the stronger packaged branch into the workspace
primary center and then re-import the missing MCP/API/docs surfaces from the
top-level `ION/` root.

The second option is to keep both roots visible for one more bounded packet
while performing a formal canonicalization decision over them before any
promotion.

The third option is to keep the split indefinitely and rely on documentation to
paper over it. I do not recommend this third option. It is the least lawful and
the most likely to recreate future center drift.

My recommendation is the second option first, then the first option if the
decision confirms the stronger packaged branch as expected.

## Decision gate 3: classify AIM-ION / AIM-OS

The third decision is whether AIM-ION / AIM-OS should currently be treated as:

- witness only,
- sibling reintegration family,
- or future dedicated lane.

My present judgment is that it should be classified as a sibling/pattern
universe with selected direct adoption candidates, not as already-integrated
canon and not as irrelevant witness residue.

## Decision gate 4: ratify the minimal foundry registry set

The fourth decision is whether the project accepts the minimal registry stack
described in the foundry operating model.

The important constraint here is that the project should ratify a small number
of high-value registries first rather than a large schema forest.

My recommendation is to start with:

- `root_manifest`
- `lineage_registry`
- `authority_registry`
- `duplicate_competition_registry`
- `canonicalization_queue`

Everything else can remain proposed until those five prove useful.

## Decision gate 5: agree the first generated export target

The fifth decision is which generated operational surface matters most first.

There are three plausible candidates:

- a root-authority bundle for human/current-branch operators,
- a browser ChatGPT compact bundle,
- or a Cursor/Codex active surface.

My recommendation is to start with the root-authority bundle first, because that
is the narrowest export that reduces the greatest current confusion.

## Open questions

The first open question is whether the top-level `ION/` MCP/API/docs surface
should be treated as a supporting bundle to merge, or whether those surfaces
should instead stay as a sibling package even after the stronger branch is
promoted.

The second open question is whether `AIM-ION` should receive its own line in the
recovery atlas equivalent to the existing major root-family profiles.

The third open question is whether the current Lane C runtime/session campaign
should consume the foundry outputs immediately or only after the first explicit
root-authority decision lands.

The fourth open question is whether tombstone/disposition templates are needed
now or only after the first retirement decisions arise in practice.

## First execution order

The cleanest first execution order now looks like this.

Step one: publish and index the foundry proposal suite so it becomes a stable
reference surface rather than ephemeral chat reasoning.

Step two: open one formal canonicalization decision over the top-level `ION/`
root and the stronger packaged branch, with retained witness value preserved for
the losing candidate center.

Step three: create the first minimal machine-readable root manifest and lineage
registry entries for the major centers already profiled in the recovery atlas.

Step four: open one explicit AIM-ION/AIM-OS classification packet that decides
whether the adjacent universe is witness-only, sibling-lane, or direct adoption
candidate by subsystem.

Step five: only after those decisions, either proceed with Lane C runtime/session
promotion work or begin the actual promotion/merge that resolves the current root
split.

## Human review points

The human operator should review at least these moments directly:

- the naming stack adoption,
- the root-authority decision,
- the AIM-ION/AIM-OS classification,
- and the first generated export bundle.

These are not routine clerical moves. They set long-horizon precedent.

## Summary

The next lawful move is not more diffuse exploration and not fresh implementation
ambition. It is a small chain of explicit decisions that turns the reintegration
program into something governable.

If those gates are passed, the canon foundry becomes real.
If they are skipped, the proposal suite will be insightful but structurally
toothless.
