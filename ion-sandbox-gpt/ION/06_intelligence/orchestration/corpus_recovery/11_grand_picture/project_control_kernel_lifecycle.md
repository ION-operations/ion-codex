# Project control kernel lifecycle

## Purpose

This file describes the smallest practical lifecycle for using the default project control kernel.

## Lifecycle

### 1. Enter the kernel
Read:
- `default_project_control_kernel.md`
- `../00_program/project_control_kernel_manifest.csv`
- `../00_program/project_control_kernel_escalation_judgment.md`

### 2. Classify and default
Use:
- `../00_program/atlas_control_card.md`
- `../00_program/future_work_control_defaults_matrix.csv`

Determine:
- question class
- primary center
- default posture
- default answer/output class
- default landing boundary

### 3. Check conflicts and gate
Use:
- `../12_status_and_conflicts/conflict_register.md`
- `../00_program/atlas_packet_gate_checklist.md`

Determine whether:
- the kernel is sufficient
- widening is required
- or reassessment is required

### 4. Choose a path
Only three paths are lawful:
- **stay inside the kernel**
- **widen explicitly into the broader atlas or preserved centers**
- **stop and reassess**

### 5. Record the start honestly
Use:
- `../00_program/atlas_gated_work_start_record_template.md`
- `../00_program/atlas_orchestration_horizon_record_template.md`

### 6. Land within the declared boundary
The work should land only where the declared landing boundary allows.
If the artifact widens in class or landing surface, the work should stop and reassess.

## Core rule

The project control kernel is not only a place to start.
It is the layer that decides whether the work may:
- remain kernel-local,
- widen lawfully,
- or must stop.
