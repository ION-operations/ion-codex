# Atlas work gating protocol

## Purpose

This protocol turns the recovery atlas from a reference library into a **future-work gate**.

Its job is to stop the project from repeating the main historical failure mode:
starting from the wrong surviving center and then inventing a local replacement for something that already exists elsewhere in the estate.

## Core rule

No new work packet, new proposal, new system claim, or new architectural judgment should proceed until it has passed through the atlas gate.

## Gate sequence

The packet gate should normally be followed by a posture selection using:
- `00_program/work_posture_selection_guide.md`
- `00_program/non_widening_work_classes.md`

Worked examples now live under:
- `00_program/gated_work_start_examples/README.md`


### Gate 1 — classify the question
Determine which class the work belongs to:
- current branch startup/routing
- runtime/API/session
- activation/manager/swarm
- meta-template/template evolution
- compact executable memory
- values/soul
- prior atlas/archaeology
- wrapper-root interpretation
- sibling-family interpretation
- genuinely new work

### Gate 2 — consult the authority matrix
Read:
- `00_program/future_work_gating_authority_matrix.csv`

That matrix tells you which preserved center should be consulted first.

### Gate 3 — read the primary authority surface
Do not start from convenience.
Do not start from the current branch by default.
Start from the authority surface that best preserves the capability in question.

### Gate 4 — check the conflict layer
Read:
- `12_status_and_conflicts/conflict_register.md`
- `12_status_and_conflicts/missing_evidence_register.md`

If the issue is already conflicted, preserve that conflict.
Do not silently resolve it just because one line feels cleaner.

### Gate 5 — decide the work posture
One of five outcomes must be chosen explicitly:
- **consult only** — no new work needed yet
- **compare centers** — more than one preserved center must be read before action
- **recover first** — evidence is still too weak; recover before acting
- **bounded extension** — new work is allowed, but only as a bounded extension of the best-preserved center
- **bridge repair** — current branch can repair/integrate, but must state what stronger historical center it is bridging from

### Gate 6 — choose the answer/output class
Use:
- `canonical_future_work_answer_output_classes.md`
- `canonical_future_work_answer_output_defaults.csv`

Future work should choose the narrowest lawful answer/output class before it begins.

### Gate 7 — choose the landing boundary
Use:
- `canonical_landing_boundary_classes.md`
- `canonical_landing_boundary_defaults.csv`

Future work should choose the narrowest lawful landing boundary before it begins.

### Gate 8 — record the starting center and orchestration horizon
Any meaningful future work should record:
- question class
- primary authority surface
- secondary support surface if any
- conflict surface if any
- chosen work posture
- chosen answer/output class
- chosen landing-boundary class
- current process position
- near / mid / far horizon
- completion condition for the current pass

## What this gate forbids

It forbids:
- defaulting to the current branch for every question
- inventing replacement systems before checking preserved lines
- flattening preserved conflicts
- treating wrappers as primary centers
- treating compact lines as automatically incomplete
- treating newer lines as automatically superior

## Current posture

At the current stage of the project, this atlas gate should be treated as the minimum lawful control layer for future planning work.

## Pass 16 packet gate extension

After consulting the authority matrix and conflict layer, future work should also satisfy the packet-gate layer:
- `future_work_packet_requirements.md`
- `bounded_extension_vs_illegal_reinvention.md`
- `atlas_gated_work_start_record_template.md`
- `atlas_packet_gate_checklist.md`

This extension exists to stop the project from treating atlas consultation as advisory only.

## Additional control surfaces

Before opening a future work-start record, also consult:
- `canonical_future_work_question_classes.md`
- `canonical_future_work_question_class_defaults.csv`

These two files reduce a recurring failure mode: starting from a distorted local framing before the atlas has even classified the kind of question being asked.


## Output and landing discipline

After posture selection, future work should also classify:
- the expected answer/output class using:
  - `canonical_future_work_answer_output_classes.md`
  - `canonical_future_work_answer_output_defaults.csv`
- the expected landing-boundary class using:
  - `canonical_landing_boundary_classes.md`
  - `canonical_landing_boundary_defaults.csv`

This prevents a correct question and a correct starting center from still producing the wrong kind of artifact or landing it in the wrong part of the organism.

## Orchestration foresight discipline

Future work should not begin on an unknown trajectory.
For work large enough to matter, also consult:
- `orchestration_horizon_and_completion_requirements.md`
- `atlas_orchestration_horizon_record_template.md`

This keeps future work dynamic but not directionless.
