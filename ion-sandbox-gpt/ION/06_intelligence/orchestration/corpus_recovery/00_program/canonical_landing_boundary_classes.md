# Canonical landing-boundary classes

## Purpose

This file defines the smallest stable set of landing-boundary classes for future work.

The goal is to stop another recurring failure mode:
future work may start from the right question, center, posture, and output class, but still drift by landing in the wrong layer of the organism.

A landing boundary answers:
- where the work may land,
- what it may mutate,
- what it must not mutate,
- and whether the work remains advisory, atlas-only, evidence-only, repair-only, or implementation-bearing.

## Canonical classes

### 1. No landing / advisory only
Use when the work should produce understanding without mutating project truth.

May affect:
- chat explanation
- optional advisory note only

Must not affect:
- law
- atlas state
- code
- profiles
- receipts

### 2. Atlas evidence only
Use when the work is recovering or clarifying historical evidence.

May affect:
- archive register
- system profiles
- capability matrix
- evidence notes
- recovery maps

Must not affect:
- active branch law
- active implementation surfaces

### 3. Conflict and missing-evidence only
Use when the correct action is to preserve contradiction or block work pending stronger recovery.

May affect:
- conflict register
- contradiction ledger
- missing-evidence register

Must not affect:
- implementation
- active law
- atlas conclusions beyond the conflict itself

### 4. Lineage and status only
Use when the work clarifies predecessor/sibling/continuation/wrapper/center status.

May affect:
- lineage register
- center-status matrix
- supersession graph
- profile status notes

Must not affect:
- implementation
- active law

### 5. Atlas operating layer only
Use when the work improves the atlas control layer itself.

May affect:
- control surfaces
- read order
- usage contract
- gating protocol
- question/output/landing classes

Must not affect:
- code
- runtime behavior
- historical evidence claims without citation

### 6. Current-branch bridge repair only
Use when the atlas already judges the work to be a narrow current-branch repair.

May affect:
- current branch docs
- current branch bridge protocols
- current branch template/protocol repair surfaces
- tightly bounded current branch code or tests if explicitly allowed

Must not affect:
- historical center claims without evidence
- wider organism interpretation beyond the repair scope

### 7. Runnable-proof layer only
Use when the work only verifies or executes an existing preserved implementation line.

May affect:
- runnable verification receipts
- proof notes
- implementation lineage evidence

Must not affect:
- implementation itself
- active law

### 8. Tightly scoped implementation surface
Use only when the atlas gate allows a bounded implementation packet.

May affect:
- narrowly scoped code
- tightly scoped tests
- local implementation receipts

Must not affect:
- wider ontology
- broad law rewrite
- uninformed replacement of stronger preserved centers

## Rule

Future work should always choose the narrowest landing-boundary class that still satisfies the question.

If a task can end in:
- no landing,
- atlas evidence only,
- conflict-only,
- lineage/status only,
- or runnable-proof only,

then it should not escalate into current-branch repair or implementation mutation.
