# Project Control Kernel Governance Outcome Classes

## Purpose

Kernel governance review should not jump from health checks directly to vague revision talk.
This file defines the smallest stable set of governance outcomes that may result from a kernel review.

## Canonical governance outcomes

### 1. Keep frozen as-is
Use when the kernel is healthy or when the observed issue is local, operator-specific, or not attributable to the kernel.

### 2. Keep frozen with clarification note
Use when the kernel remains sound but a persistent reading ambiguity should be clarified without changing defaults or structure.

### 3. Keep frozen with defaults review queued
Use when the kernel remains broadly sound but a patterned defaults mismatch should be scheduled for targeted defaults review.

### 4. Enter thaw for clarification-only revision
Use when repeated ambiguity is attributable to the kernel text itself and cannot be responsibly handled as a local note.

### 5. Enter thaw for defaults correction
Use when the defaults layer is repeatedly routing work into patterned dead ends or wrong-center starts.

### 6. Enter thaw for new control primitive proposal
Use when repeated work-start or governance history shows a control necessity that is genuinely absent from the kernel.

### 7. Consider freeze rollback
Use only when the frozen kernel is shown to be structurally inadequate in a way that cannot be corrected by clarification, defaults correction, or a narrowly bounded new primitive.

## Governance default

The default bias is toward the smallest lawful governance outcome.
Do not escalate from review to thaw merely because a stronger rewrite feels cleaner.
