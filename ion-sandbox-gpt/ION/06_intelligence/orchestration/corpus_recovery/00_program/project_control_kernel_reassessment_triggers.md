# Project Control Kernel Reassessment Triggers

## Purpose

This file defines when work should **stop and reassess** instead of remaining inside the kernel or continuing to widen.

The project historically drifted because work could continue under plausible momentum even when:
- the question was framed badly,
- the wrong center was being consulted,
- the wrong output was being produced,
- or widening had become self-justifying.

---

## Core rule

A stop-and-reassess trigger is not failure.
It is a lawful control event.

---

## Canonical reassessment triggers

### T1 — Question-class instability
The work no longer fits the original question class.

### T2 — Center insufficiency
The selected primary center clearly cannot answer the actual question without a new center comparison.

### T3 — Output mismatch
The work is drifting toward a broader or more mutative output class than originally declared.

### T4 — Landing-boundary mismatch
The work is about to land outside its declared landing boundary.

### T5 — Horizon mismatch
The current pass can no longer state:
- where it is,
- what near/mid/far horizon means,
- or what “complete enough for this pass” is.

### T6 — Conflict deepening
The conflict layer is stronger than expected and cannot be honestly flattened inside the current pass.

### T7 — Reinvention risk
The work is beginning to invent a replacement rather than recover, compare, or bridge.

### T8 — Return impossible
Widened work can no longer state how it would return to the control kernel.

---

## Reassessment outcomes

When a trigger fires, the work should return with one of these outcomes:
- reclassify question
- compare centers
- recover first
- narrow landing boundary
- demote output class
- halt current widening
- escalate to missing-evidence entry

---

## Default rule

If two or more reassessment triggers fire at once, the default should be:
**stop and return to the kernel immediately.**
