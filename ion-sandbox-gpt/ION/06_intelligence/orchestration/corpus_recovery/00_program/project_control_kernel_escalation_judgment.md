# Project control kernel escalation judgment

## Purpose

This judgment answers the next practical question after establishing the default project control kernel:

**When is the kernel enough by itself, and when must work widen beyond it or stop and reassess?**

## Current judgment

The default project control kernel is now strong enough to govern **most work-start decisions**, but it should still be treated as a **gate and routing kernel**, not as a self-sufficient total organism substitute.

In practice, future work should move through three possible states:

1. **Stay inside the kernel**
2. **Widen into the broader atlas or preserved centers**
3. **Stop and reassess before proceeding**

## Stay inside the kernel when

Remain inside the default project control kernel when all of the following are true:
- the question class is clear
- the defaults row is clear
- the primary preserved center is already known
- preserved conflicts do not require direct comparison
- the intended output class is narrow and lawful
- the intended landing boundary is narrow and lawful
- the current pass has a clear completion condition

This is the ideal case for:
- consult-only work
- simple conflict-preserving judgments
- narrow atlas updates
- narrow non-widening advisory outputs

## Widen beyond the kernel when

Widen into the broader atlas or preserved centers when any of the following becomes true:
- the defaults row points to a center that must be read directly
- preserved conflicts require direct comparison between centers
- the task is recover-first and deeper evidence is required
- the task is a real bridge repair and must compare against stronger historical centers before landing
- the task involves a genuinely absent capability claim that must be proven before bounded extension

Widening should be explicit, named, and bounded.
It should never happen merely because the kernel feels too small or because local invention is convenient.

## Stop and reassess when

Stop and reassess before proceeding when any of the following becomes true:
- the question class cannot be assigned cleanly
- the landing boundary is unclear
- the task starts producing a wider artifact class than originally declared
- the work begins implying new architecture or new ontology without strong absence evidence
- the current process position, near/mid/far horizon, or completion condition cannot be stated honestly
- the non-reinvention claim becomes weak

This is important because one of the main ION failure modes is continuing a plausible workflow on an unknown trajectory.

## Meaning for the project

The default project control kernel is now best understood as:
- a **stable work-start kernel**
- a **routing and discipline kernel**
- a **widen/stop decision kernel**

It is not a replacement for:
- preserved centers
- deeper recovery evidence
- or wider atlas consultation when genuinely required

## Added return and reassessment layer

The escalation layer is now strengthened by three companion surfaces:
- `project_control_kernel_return_conditions.md`
- `project_control_kernel_reassessment_triggers.md`
- `project_control_kernel_escalation_record_template.md`

This means the kernel now governs not only whether work may widen, but also:
- when widened work should return,
- when it should stop and reassess,
- and how that escalation should be recorded.
