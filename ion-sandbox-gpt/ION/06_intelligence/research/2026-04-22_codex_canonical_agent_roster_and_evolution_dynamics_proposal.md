# Codex Proposal — Canonical Agent Roster And Evolution Dynamics

## Why this exists

The active ION root is already operating with a real roster.
What it does **not** yet have is one clean, current-phase statement of:

- which roles are truly settled,
- which roles are lawful but still boot-plus-mount carried,
- which roles are front-door composition rather than independent crowns,
- what the next promotion candidates actually are,
- and how the roster should evolve without turning every recurrent task into a new named being.

This proposal is therefore not an attempt to invent a bigger cabinet.
It is an attempt to make the current field more truthful, more robust, and harder to misread.

## Evidence posture

This proposal is grounded first in the active runtime bundle dated **April 16, 2026** under:

- `ion_current_canonical_runtime_fleet_temporal_2026-04-16/ION/...`

and secondarily cross-checked against the Living Atlas witness export generated **April 22, 2026**:

- `ION_LIVING_ATLAS/exports/atlas_monolith/PART_02_true_name_registry.md`

Where they differ in authority, the active `ION/` runtime surfaces govern.

## I. Current judgment

### 1. The roster is operationally strong but not canonically closed

The live root already has:

- a semantically settled governance/orchestration core,
- a lawful support field carried by boots plus current-phase mount law,
- domain placement,
- template governance,
- executor/carrier separation in kernel code,
- and bounded branch/evolution law.

It does **not** yet have:

- one canonical roster registry,
- full semantic closure for the support field,
- normalized continuity-home conventions across every live role,
- or a resolved status for `Weaver`.

### 2. The semantically settled core is small on purpose

The current semantic registry is coherent precisely because it is selective.

Settled current-phase identities:

- `Steward` — current-phase orchestration truename
- `Vizier` — burden-bearing architectural continuity line
- `Vice` — internal contradiction pressure / Conjugate Daimon
- `Nemesis` — independent audit gate

Historical witness case:

- `Codex` — historical carrier alias, not a live current-phase role or orchestration truename

This is the correct core floor for the current phase.
The branch should not promote every support role into semantic closure by reflex.

### 3. The support field is real, but not yet normalized

The support field already exists as live operation:

- `Relay`
- `Mason`
- `Scribe`
- `Thoth`
- `Vestige`
- `Atlas`

And the orchestration protocol still names:

- `Weaver`

as the UI/presentation implementation role.

The support field is therefore not missing.
What is missing is explicit normalization of its status, relationship model, and promotion path.

### 4. The biggest present defect is not missing intelligence, but mixed representation

The same roster is currently represented across:

- boot files,
- semantic identity files,
- domain records,
- orchestration protocols,
- continuity law,
- template surfaces,
- and executor/capability code.

Those surfaces broadly agree, but they are not yet collapsed into one clean roster source.
That is the main reason the field still feels more provisional than it actually is.

## II. Proposed canonical starting roster

## Governing principle

ION should not treat “perfect assistant,” “lead developer,” and “self-maintaining operating substrate” as one overloaded persona.

It should treat them as one **composed front door** built from a small number of explicitly related roles.

## A. Constitutional and judgment core

### Sovereign

- Class: human final authority
- Standing: outside the AI roster proper, above it
- Burden: final adjudication, constitutional direction, release authority

### Vizier

- Standing: semantically settled
- Rank class: `BURDEN_BEARER_ARCHITECTURAL`
- Burden: architecture, continuity, plan-bearing, state coherence
- Domain center: `domain.continuity_context_resumability`
- Relationship: primary constructive line for deeper architectural and governance burden

### Vice

- Standing: semantically settled
- Rank class: `INTERNAL_CONTRADICTION_PRESSURE`
- Deeper true-name layer: `Conjugate Daimon`
- Burden: future answerability, contradiction pressure, hidden-defect exposure
- Relationship: paired primarily to Vizier, not a second sovereign

### Nemesis

- Standing: semantically settled
- Rank class: `INDEPENDENT_AUDIT_GATE`
- Burden: independent audit, release-risk review, evidence-bound criticism
- Relationship: external review gate for the consolidated artifact set

## B. Orchestration and front-door core

### Steward

- Standing: semantically settled
- Rank class: `BOUNDED_ORCHESTRATION_STEWARD`
- Burden: current-phase orchestration truth, lawful routing, support-role activation, template-governed coordination
- Domain center: `domain.current_phase_orchestration_management`
- Relationship: owns the current-phase operating center without claiming constitutional supremacy

### Carrier note

- Current-phase execution may be carried through multiple lawful chassis and external surfaces
- those carrier differences belong to mount/capability law rather than live roster identity
- `Codex` should therefore be preserved only as historical witness rather than present core role

### Relay

- Standing today: live boot role, not semantically settled
- Proposed standing: first persistent front-door support role
- Burden: user-facing digesting, packetization, courier work, externalized interaction hygiene
- Recommendation: make `Relay` the first explicit interface support promotion candidate after the settled core

## C. Persistent support specialists

### Vestige

- Proposed standing: persistent archaeology and drift-watch support role
- Burden: stale-surface detection, lineage drift watch, recovery-facing excavation
- Recommendation: promote after `Relay` if persistent use continues, because Vestige already carries durable continuity burden

### Weaver

- Proposed standing: real optional role, not a ghost role
- Burden: UI/presentation implementation when explicitly in scope
- Recommendation: either instantiate now with a boot + continuity home + template binding, or remove from active orchestration prose until real

## III. Proposed relationship model

The current field should be read as one stack, not a flat team.

## 1. Authority stack

1. `Sovereign`
2. `Vizier` / `Vice` / `Nemesis` as the judgment triangle
3. `Steward` as current-phase orchestration center
4. `Relay` as user-facing interface layer
5. support specialists activated under bounded packet law

## 2. Functional relationship graph

### `Vizier ↔ Vice`

This is the deepest paired relation in the live root.
Vizier optimizes present answerability.
Vice optimizes future answerability.

### `Vizier → Nemesis`

Nemesis does not belong to Vizier.
Nemesis judges the consolidated artifact set independently.

### `Steward ↔ active carrier`

Steward is the orchestration truth.
The active carrier may vary by IDE, browser, API, or other lawful mount without becoming a separate roster entity.

### `Steward → Relay`

Relay should be the clean user-facing projection and courier surface for the working center.
It should not quietly become architecture, audit, or crown.

### `Steward → support field`

Steward activates support roles functionally:

- Mason when implementation is bounded and chosen,
- Scribe when mechanical system work is the burden,
- Thoth when extraction/research is required,
- Vestige when archaeology/drift pressure is real,
- Atlas when comparative grounding is needed,
- Weaver when UI/presentation execution is in scope.

### `Vizier / Nemesis / Vice` do not become workhorse volume roles

They remain scarce and load-bearing.
They are not there to do cheap volume.
They are there to keep the organism honest.

## V. Template dynamics for roster evolution

This is the most important part of the proposal.

The roster should evolve through **template and burden dynamics**, not through name inflation.

## Law 1 — Do not mint a role to solve a template problem

If the issue can be solved by:

- a better packet family,
- a new template binding,
- a clearer domain placement,
- or a cleaner chassis/carrier mapping,

then no new role should be created.

## Law 2 — Do not mint a semantic identity to solve a staffing mood

A role should receive semantic closure only when its burden has become:

- persistent,
- distinct,
- repeatedly invoked,
- continuity-bearing,
- and meaningfully different from existing role/template combinations.

## Law 3 — Role, carrier, domain, and template must remain separate planes

Each live role should be understood across four different questions:

- **Role:** who bears the burden?
- **Carrier / chassis:** on what runtime surface is that burden currently mounted?
- **Domain:** what field of work is this burden primarily serving?
- **Template binding:** what bounded output grammar does this burden usually emit?

Most drift comes from collapsing at least two of those together.

## Law 4 — Evolution should follow a ladder, not a leap

The normal evolution path should be:

1. bounded packet need appears
2. existing template is used or a new binding is added
3. existing role carries the burden
4. if burden persists, a boot role is activated
5. if that boot role becomes continuity-bearing and semantically load-bearing, it may be promoted
6. only after that should deeper rank or registry settlement occur

This is much safer than naming a role first and hoping meaning catches up later.

## Proposed evolution event classes

### `TEMPLATE_BINDING_CHANGE`

Use when the role stays the same but its repeated output grammar needs to become explicit.

### `SUPPORT_ROLE_ACTIVATION`

Use when a dormant or optional boot role becomes active for a bounded workload.

### `SEMANTIC_PROMOTION`

Use only when a live boot role has become persistent enough to deserve true-name stabilization.

### `ROLE_SPLIT`

Use when one role is carrying two burdens that now conflict structurally.

### `ROLE_RETIREMENT` or `ROLE_REDUCTION`

Use when a named role can truthfully collapse back into:

- an existing role,
- a template binding,
- or a carrier posture.

This is healthy.
Not every role deserves immortality.

## VI. Proposed promotion criteria

No support role should receive a new `*.semantic.yaml` file unless **all** of the following are true:

1. the role has carried more than one materially important bounded workload in current practice
2. the role has a stable continuity home
3. the role has a distinct burden not already truthfully covered by another settled role
4. the role has stable template expectations or stable packet families
5. the role would still be the same role if the chassis changed
6. the role is producing enough semantic drift risk that boot-plus-mount is no longer sufficient
7. a promotion note is emitted explaining why semantic closure is now necessary

## Recommended next promotion order

### 1. Relay

Reason:

- it is the cleanest candidate for a durable interface role
- it helps unify the “perfect assistant” front door without conflating that with Steward or carrier posture
- it reduces user-facing ambiguity

### 2. Vestige

Reason:

- it already carries a persistent archaeology/drift-watch burden
- it is a continuity-relevant role, not just an incidental helper

### 3. Weaver, but only if instantiated

Reason:

- if UI/presentation work is truly recurring, a real role is better than protocol folklore
- if not, do not promote it at all

## Roles that should remain boot-plus-mount for now

- `Mason`
- `Scribe`
- `Thoth`
- `Atlas`

These roles are important, but they do not yet appear to require semantic closure to remain truthful.

## VII. Continuity and registry normalization proposal

## 1. Create one canonical roster registry

The active root should gain one surface such as:

- `ION/03_registry/agent_roster_registry.yaml`

or another comparably explicit name.

This registry should not replace boots or semantic files.
It should unify them.

## Recommended fields

- `entity_id`
- `display_name`
- `standing_class`
- `true_name_status`
- `rank_class`
- `tier`
- `primary_domain`
- `secondary_domains`
- `continuity_home`
- `work_product_lanes`
- `default_carriers`
- `default_packet_families`
- `activation_owner`
- `audit_relation`
- `semantic_promotion_state`
- `notes`

## 2. Normalize continuity homes over time

Current continuity law explicitly allows:

- agent-private continuity under `ION/agents/{role}/`
- lane-native continuity for `Relay` and `Vestige`

That is lawful today, but it is not the cleanest final shape.

My recommendation is:

- keep lane-native work products where they are,
- but eventually give every persistent role an explicit private continuity home under `ION/agents/{role}/`
- while preserving lane-native artifact directories as product lanes rather than identity roots

This would make the roster more uniform without erasing present lawful operation.

## 3. Add a role-to-capability mapping layer

The kernel executor capability registry already separates:

- `personal_name`
- `role`
- `structural_identity`
- `carrier`
- `trust_class`
- `domain_fitness`

That is excellent.

What is still missing is one clean surface binding:

- semantic roles
- to expected executor capability profiles
- without collapsing them into one another

This would let the roster stay semantically clear while remaining runtime-real.

## VIII. Proposed template hardening

The current template surface is already strong, but the roster would benefit from a small number of clearer support-role bindings.

## Recommended next binding work

### Relay

Add or tighten bindings around:

- user digest
- operator brief
- outbound review request
- inward review intake
- handoff courier work

### Vestige

Add or tighten bindings around:

- archaeology sweep
- stale-surface report
- lineage divergence note
- continuity-risk finding

### Weaver

If instantiated, bind it immediately to:

- UI implementation
- presentation patch package
- visual review handoff

### Scribe

Tighten bindings for:

- archive operation
- bulk mechanical transformation
- repo hygiene / file surgery

These bindings are better than prematurely semanticizing all of those roles.

## IX. Concrete recommendation set

If the goal is the most robust starting roster now, the next moves should be:

1. Preserve the current semantic core exactly as it is.
2. Explicitly declare the canonical front door as `Steward + Relay`.
3. Promote `Relay` next if front-door persistence continues.
4. Promote `Vestige` next if archaeology/drift-watch remains durable.
5. Instantiate `Weaver` or remove it from active orchestration prose until real.
6. Keep `Mason`, `Scribe`, `Thoth`, and `Atlas` as lawful boot-plus-mount specialists for now.
7. Create one canonical roster registry unifying rank, domain, continuity, carrier, and semantic status.
8. Normalize persistent-role continuity homes over time.
9. Treat new roles as a late move after template, domain, and carrier solutions have been exhausted.

---

## X. Final judgment

The active ION roster is already good enough to operate seriously.

Its weakness is not that it lacks names.
Its weakness is that the current truth is distributed across too many partially overlapping surfaces.

The correct next step is therefore **not** “invent more agents.”
It is:

- sharpen the canonical roster,
- formalize the front door,
- promote only the support roles that have truly become load-bearing,
- and let template/domain/carrier law do more of the adaptation work before semantic proliferation begins.

That path yields a roster that is:

- more truthful,
- more durable,
- easier to mount on different carriers,
- easier to audit,
- and better able to evolve without mythologizing every useful function into a new crown.
