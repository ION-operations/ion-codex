# Continuity Recovery Conditions

> This companion note is intentionally separate from the continuity law.
> The law defines what is true.
> These conditions define what must still be repaired before continuity-sensitive
> release, clone scaling, or automation promotion can be treated as safe.

## Recovery conditions that remain active

### R1 — Clone scaling remains blocked

No clone scaling until one end-to-end lawful cycle is demonstrated:

task arrival → private continuity update → signal emission →
projection generation/reconciliation → successful fresh-chat restart from continuity

### R2 — Root-as-source language must be corrected

Any active document, boot, or role instruction that still treats shared
root MINI/CAPSULE/STATUS as universal raw source continuity must be patched.

This condition has narrowed materially:

- core builder/audit boots are now largely corrected
- root projections now largely teach the right model

The remaining pressure is concentrated in:

- `ION/PLAN.md`
- `RELAY.boot.md`
- `VESTIGE.boot.md`
- legacy daimon boot artifacts and any stale supervisor guidance

### R3 — Automation remains non-authoritative until shadow proof

No continuity compiler or projection builder becomes primary until it matches
manual source continuity for at least one full clean work cycle.

### R4 — Continuity-sensitive release requires visible Daimon participation

For covered artifact classes, continuity-sensitive release remains blocked if
the Daimon lane has no visible artifact for Nemesis to audit against the consolidated set.

### R5 — No runtime or phase claims beyond current reality

No claim should imply that the unified `ION/` runtime or authoritative compiled
context pipeline already exists until the relevant implementation stages are landed.

### R6 — Role continuity matrix must be made explicit

This condition has been partially addressed by:

- `ION/06_intelligence/roundtable/continuity_crisis/proposals/2026-04-03_role_continuity_matrix.md`

What remains is ratification and edge-case normalization.

The system still needs a single accepted table for each active role stating:

- whether it has a private MINI
- whether it has a private CAPSULE
- whether the boot reads private continuity first
- whether it may write root projections
- whether it may emit task files
- its continuity class (`agent-private`, `lane-native`, `legacy adapter`)
- whether the minimum source continuity actually exists yet

## Immediate implementation sequence after ratification

1. Patch the remaining edge-case boots to align with the law.
2. Finish normalizing private continuity roots and edge-case lane-native roles.
3. Keep the physical bus honest and verify task/signal flow.
4. Patch the major templates with explicit manual continuity update obligations.
5. Run a stronger second lawful work cycle and record source/projection drift.
6. Only then promote shadow automation or widen clone scaling.

## Operational interpretation

Ratification of the continuity law should begin the repair sequence,
not be mistaken for proof that recovery is complete.
