# ROOT INVARIANT RESTORATION PROTOCOL

**Status:** Current-phase operational proposal  
**Date:** 2026-04-24  
**Authority posture:** A3 until reviewed  
**Purpose:** Define how ION restores a lost, buried, or fragmented root invariant without laundering it into instant final canon.

---

## 1. Definition

A root invariant is an organism-level statement that organizes many downstream systems.

Examples:

```text
ION is a living context graph operated by lawful agents through templates.
ION is not trying to make one AI bigger; it is trying to make context itself into an operating substrate.
A valid completed template file is an automation event surface.
```

Root invariants are dangerous because they are powerful. If correct, they clarify everything. If premature or false, they distort everything.

---

## 2. Restoration posture

A recovered root invariant must begin as:

```text
PROVISIONAL_RESTORED_ROOT_INVARIANT
```

It may not become ratified law merely because it is elegant, useful, or emotionally clarifying.

---

## 3. Required evidence

A restoration packet must include at least three evidence classes where possible:

```text
origin evidence
donor-line evidence
current-root evidence
runtime enactment evidence
contradiction evidence
terminology drift evidence
loss/recovery evidence
```

If fewer than three are available, the packet must say why.

---

## 7. Non-loss clauses

Root invariant restoration must not:

1. erase older formulations that remain true;
2. overwrite current runtime truth;
3. collapse governance view and structural-substrate view;
4. claim full enactment where only doctrine exists;
5. ignore graph.py/current implementation mismatch;
6. bypass self-modification review;
7. update public product language without approved-context review.
