## Purpose

ION must not mistake the existence of doctrine, temporal systems, context systems, templates, tests, or plans for operational connection. This protocol defines the V104 audit discipline for distinguishing:

```text
present surface      = file/module/protocol exists
wired surface        = a live runtime path calls or consumes it
enforced surface     = execution is blocked or classified when the rule fails
visible surface      = cockpit/operator state exposes the result
packaged surface     = carrier/release zips obey the result
```

The failure class discovered between V102 and V103 was not that ION lacked temporal/context doctrine. The failure class was that several advanced surfaces existed without being universally enforced across carrier packaging, active state regeneration, and autonomous-loop execution.

## Audit law

A V104 operational audit must check at least:

1. shell-root invariant;
2. active state files;
3. active carrier work-cycle template binding;
4. active context-window plan freshness;
5. role context-system completeness;
6. local autonomous-loop survival modules;
7. temporal/context lifecycle enforcement;
8. hot/warm/cold context lifecycle posture;
9. worker-adapter boundary;
10. long-horizon operational roadmap presence.

## Repair law

The audit may perform only conservative reconnection repairs that do not delete, archive, compress, or rewrite evidence. V104 permits:

- creating a missing active work-cycle template when an active packet already points at that path;
- refreshing stale `exists` flags when the referenced files demonstrably exist;
- filling an empty role context-system card with bounded non-production role law;
- writing audit reports, manifests, locks, and open-work registers.

V104 does not permit:

- production authority;
- destructive cleanup;
- cold archive movement;
- external worker activation;
- MCP/API worker execution;
- release-candidate ratification.

## Long-horizon orchestration law

ION is not fully operational until the following gates are closed in order:

```text
G1 operational truth audit and active-state repair
G2 lifecycle-aware carrier/release packager
G3 front-door Relay/Steward/Persona runtime path
G4 bounded worker-adapter interface
G5 template graph writeback diff/review/commit path
G6 schedule/context-delta timeline integration
G7 cockpit/JOC operator proof surface
G8 release verifier and production namespace repair
G9 external carrier/API/MCP execution under gates
G10 production-readiness adversarial audit
```

No later gate may bypass context proof, template/action proof, Steward integration, human gate state, lifecycle packaging policy, or visible stop reason.
