# CANON PROMOTION AND RATIFICATION PROTOCOL

**Status:** Current-phase operational proposal  
**Date:** 2026-04-24  
**Authority posture:** A3 until reviewed  
**Purpose:** Define how provisional doctrine, protocol, template, semantic identity, role, or registry objects become canon.

---

## 1. Promotion classes

```text
PROMOTE_TO_EXPLANATORY_CANON
PROMOTE_TO_OPERATIONAL_PROTOCOL
PROMOTE_TO_TEMPLATE_LAW
PROMOTE_TO_REGISTRY_CANON
PROMOTE_TO_RUNTIME_ENACTED_SURFACE
PROMOTE_TO_CONSTITUTIONAL_INTERPRETIVE_INVARIANT
PROMOTE_TO_CONSTITUTIONAL_LAW
```

Each has different evidence requirements.

---

## 2. Required promotion evidence

All promotions require:

```text
source object
authority posture
reason for promotion
donor comparison result
contradiction review result
affected surfaces
required tests
receipt
rollback or supersession rule
```

D4-D5 promotions require re-expression validation.

Runtime promotions require tests or smoke evidence.

---

## 3. Canon does not mean eternal

Canon may later be:

```text
amended
superseded
retired
historicalized
quarantined
split
merged
```

But only through explicit receipt-backed action.

---

## 4. No silent promotion

A file appearing in the current root does not make it canon.

A generated encyclopedia sentence does not make it canon.

A passing test does not make doctrine canon.

A runtime surface does not make public product wording canon.

Promotion requires a promotion record.
