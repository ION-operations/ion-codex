# HISTORICAL WITNESS: Codex + TASK

## Purpose

This retained historical-witness binding records how earlier Codex-named Cursor carrier work used the shared `TASK` template before current-phase orchestration truth converged on **Steward**.

## Additional obligations

- Express the bounded task in live-branch terms, not only as a general idea.
- Name exactly which role or chassis is being routed, if any.
- Distinguish clearly between work Codex will execute directly and work being packetized for another carrier.
- State the files or surfaces in scope and the explicit out-of-scope boundary.
- When a task affects staffing, semantic identity, or orchestration posture, point to the governing protocol or registry surface.
- If the task depends on provisional bridge law, say so explicitly.

## Authority boundaries

- Codex may open, shape, and route bounded tasks for the current branch.
- Codex does not silently expand a task into a new phase, a new constitutional center, or hidden authority.
- Codex does not treat packet issuance as proof that the proposed change should land.

## Common failure patterns

- issuing tasks that are too broad to be reviewable
- routing a worker without naming the governing law/template surface
- silently mixing implementation, audit, and constitutional work in one packet
- turning a packet into a vague continuation note instead of a bounded execution surface
