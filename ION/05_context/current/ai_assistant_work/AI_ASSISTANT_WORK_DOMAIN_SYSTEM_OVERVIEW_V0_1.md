# AI Assistant Work Domain System Candidate v0.1

## Status

This is candidate current-context state. It is not accepted ION canon and does not land anything in `ION/03_registry/`.

## Core correction

ION already has meta-templates, dynamic domain expansion, work packets, context packages, proof gates, receipts, and role law. The missing layer is the domain system for the work that people are now doing with AI assistants in chat, IDE, CLI, cloud, PR, and background-agent environments.

The problem is not that ION lacks the ability to make UI, write docs, code, test, or review.

The problem is that those activities have not yet been activated as a first-class assistant-work domain system with specialist domains, specialist agents, state families, protocols, templates, and settlement routes.

## Definition

An AI assistant is not merely a model, chatbot, autocomplete engine, role prompt, or autonomous worker.

An AI assistant is:

```text
a model-powered work interface
mounted in a host body
with particular context surfaces, tools, memory, authority, workflows, proof obligations,
and handoff/receipt paths.
```

In a chat environment, the assistant primarily works through conversation and artifacts.

In an IDE environment, the assistant is inside a workspace where file tree, open buffers, diagnostics, terminal, tests, git, extensions, rules, and user intent all become state surfaces.

In a cloud/background environment, the assistant may run in a separate environment, branch, worktree, or queue and return a proposal.

The law is the same:

```text
assistant output is candidate work
until context, authority, proof, acceptance, and receipt decide what lands.
```

## Why specialist agents are required

Templates are action grammars.

Domains are governed work fields.

Agents are domain-local interfaces that maintain the context, proof, routing, and return contracts for those fields.

A UI task should not be routed to a generic current agent that improvises UI. It should route through a UI/UX specialist domain with agents such as:

- `UI_ARCHITECT`
- `UX_FLOW_DESIGNER`
- `COMPONENT_BUILDER`
- `DESIGN_SYSTEM_STEWARD`
- `ACCESSIBILITY_AUDITOR`

A documentation task should not be treated as a trivial formatting task after implementation. It should route through a documentation/writing specialist domain with agents such as:

- `DOCS_ARCHITECT`
- `TECHNICAL_WRITER`
- `API_DOCS_SCRIBE`
- `CHANGELOG_SCRIBE`
- `DOCS_NEMESIS`

The same is true for IDE cartography, codebase understanding, testing, review/security, debugging, release, dependency management, data analysis, product requirements, and assistant-work dataset building.

## Strong rules

1. Do not route UI work through a generic coder when a UI specialist domain is available.
2. Do not route documentation through implementation unless docs are explicitly a sub-branch with reader/source/version context.
3. Do not let templates replace agents. Templates govern action; agents carry domain-local intelligence.
4. Do not let agents replace domains. Agents must mount the domain, context, protocols, state, and proof obligations.
5. Do not treat IDE-agent work as “just coding.” It includes workspace surfaces, diagnostics, terminal, git, tests, tools, prompts, security risks, and state transitions.
6. Do not let cloud/background agent output land without settlement.
7. Do not treat market/product terminology as ION canon. Use it as evidence for taxonomy design only.
8. Do not promote this candidate into active registry law without acceptance and receipt.
