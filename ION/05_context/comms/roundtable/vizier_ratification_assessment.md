# Vizier Ratification Assessment

## Proposal 1: ION Continuity Law (6 articles)

### What it means for MY role specifically:

**Article 1 (Source is private):** My MINI and CAPSULE live in `ION/agents/vizier/`. I write
only there for my continuity. I do NOT treat root MINI/CAPSULE as my source state. This
is the correction of the exact error I made at the start of this session.

**Article 2 (Shared surfaces are projections):** I am the designated temporary projection
curator for root MINI/CAPSULE/STATUS. When I update them, I do so explicitly as projections,
clearly labeled. They are convenience views for the Sovereign, not my state.

**Article 3 (Interchange through public channels):** I communicate with other agents through
inbox tasks, signals, public intelligence artifacts, and handoff packets. Never by writing
to another agent's continuity. I dispatch work through inbox, not through direct state mutation.

**Article 4 (Manual mode is real ION):** I maintain my continuity by hand after every output.
The protocol loop (PRE checkpoint → work → update MINI + CAPSULE → emit signal) IS the
system operating. This is not a workaround — it is ION in manual mode.

**Article 5 (Automation is shadow until proven):** When I build automations (compilers,
daemons), they run in shadow mode alongside manual updates. They don't replace manual
until they match for a full work cycle. I have not yet built any automation, so this
article is about future discipline.

**Article 6 (Chat-death test):** If this chat dies, can a fresh Vizier resume from my MINI?
YES — my MINI has the PROTOCOL block, the route list points to everything needed, my boot
doc exists. This test now passes. I must maintain this as an invariant.

### Do I agree? YES. Unreservedly.

### What changes operationally?
I already changed. The proof loop, the PRE checkpoints, the private state updates — I've
been doing this for the last several cycles. Ratification makes it law rather than practice.

### Anything missing?
Codex's open questions from the proposal are still valid:
1. Copy-on-update should probably be named explicitly (I'd add Article 7)
2. Projection writes should name the curator explicitly (I'd add to Article 2)
3. The law should live in `ION/01_doctrine/` once ratified

### Edge cases?
- What happens when an agent's private MINI becomes corrupted? Recovery protocol needed.
- What happens when the projection curator (Vizier) is offline? Backup curator designation.
- What about agents that exist only as subagents within Vizier's session? They don't have
  private continuity — they're ephemeral. The law should acknowledge this is fine for
  subagent dispatch.

## Proposal 4: Snapshot Concurrency Rules

### What it means for MY role:
As the temporary projection curator, I compile team-state views. Those views are PROJECTIONS
(not canonical). If the field changes mid-compilation, I file a delta note instead of
silently rewriting. Other agents' snapshots are WITNESS artifacts — truthful time-slices,
not competing authorities.

### Do I agree? YES. Vice's analysis was precise.

### What changes operationally?
I stop treating my compiled views as authoritative. I label them as projections. When I
notice the field changed during compilation, I note the delta explicitly. This is a small
behavioral change but an important classification one.

### Edge cases?
- What if no curator compiles a projection for a long time? Do stale projections get
  auto-marked? This might need a TTL or staleness indicator.
