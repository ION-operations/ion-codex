# Codex End-to-End Review and Kernel-Router Transition

## Why this exists

The roundtable has largely converged the continuity floor, but the total operating picture
is still split between:

- what the active `ION/` root actually contains,
- what several architecture/protocol files still assume,
- and what the team can now afford to run.

The new hard constraint is operational:

> On 2026-04-03, the team burned roughly $500 of Cursor API usage in about 7 hours.

That spend was not pointless. It bought real convergence and real system appearance.
But it means the previous default posture of many simultaneously active premium-chassis
agents is no longer the safe baseline.

The operating shift this implies is:

> Codex should now be treated as the default sequential kernel router for the active
> IDE-native ION build, with other chats and lighter agents used selectively rather than
> continuously.

This note is not a doctrine ratification artifact.
It is an operational truth package.

---

## Core judgment

ION now has a credible continuity law core.
What it does **not** yet have is a fully restored surrounding substrate.

Specifically:

1. The active root now has **continuity law, role boots, and a minimal doctrine floor**,
   but not a fully restored constitutional doctrine stack.
2. The active root began this pass without a live template tree; a minimal `07_templates/`
   stack was restored during this pass, but the fuller doctrine-bound template layer still
   does not exist.
3. The active root has **private agent lanes**, but not compiled context infrastructure.
4. Several architecture documents still assume a broader, more parallel, more premium-heavy
   runtime than the team should default to after today's spend.

So the right move is not "pretend the grand architecture is fully landed."
The right move is:

- preserve the converged law,
- reduce stale protocol teaching,
- and shift actual operation to a lower-burn, Codex-centered sequential routing model.

## Template reality review

At the start of this pass, the active root did **not** contain a first-class
`07_templates/` tree.

A minimal live template stack has now been created, but the system is still operating on a
transitional stack composed of:

- frontmatter conventions
- role-specific boot instructions
- task packet format in `05_context/inbox/`
- signal format in `05_context/signals/`
- proposal / audit / research markdown structure
- minimal local templates in `ION/07_templates/`
- external template references in `SOS-OPUS/07_templates/`

This is one of the most important truths in the whole system right now.

ION keeps talking like templates are first-class constitutional machinery.
In the active root, they are now minimally present but still only partially restored.

### Immediate practical rule

Until the full template stack is rebuilt, every bounded work unit should explicitly state:

- output class
- destination path
- required sections or schema
- review path
- visibility expectations

In other words:

> If the template is not physically present, the task packet or boot must carry the
> missing template obligation in plain view.

## Transitional operating model

## Codex as sequential kernel router

For the current phase, the default runtime should be:

- one high-capability Codex session
- dynamic onboarding into the relevant role context as needed
- sequential execution of routing, synthesis, bounded implementation support, and drift watch
- selective activation of other agents only where the cost/benefit is clear

This does **not** mean Codex becomes hidden sovereign, hidden Vizier, hidden Nemesis,
or hidden Relay.

It means:

Codex becomes the **active kernel router surface** for a budget-constrained phase.

### What that means operationally

1. **Intake**
   Codex receives the user or operator request.

2. **Classification**
   Codex determines whether the task is primarily:
   - architecture
   - audit
   - relay
   - research
   - implementation support
   - archaeology
   - or multi-domain

3. **Dynamic role onboarding**
   Codex reads only the role boot, continuity lane, and protocol surfaces needed for that task.

4. **Sequential execution**
   Codex performs the next bounded unit directly instead of defaulting to parallel dispatch.

5. **Selective auxiliary use**
   Composer 2 or other lower-cost roles should be activated only for:
   - mechanical volume work
   - repetitive file operations
   - bulk evidence sweeps
   - clearly bounded implementation slices

6. **Visible provenance**
   If Codex acts across role boundaries, the artifact must say what it is:
   - Codex analysis
   - Codex-compiled backfill
   - Codex patch package
   - Codex witness snapshot

7. **No hidden role theft**
   Codex should not silently overwrite another role's private continuity or pretend to be
   that role's authentic internal voice unless the operator explicitly instructs it to do so.

---

## Dynamic onboarding rule set

When Codex needs to operate against another role's domain, the load order should be:

1. governing directive or user request
2. relevant global law/protocol surface
3. target role boot
4. target role private MINI/CAPSULE if the task genuinely depends on them
5. relevant task packet, signal, or active artifact set
6. bounded execution
7. Codex continuity update + public signal

This keeps context loading proportional instead of loading the whole field every turn.

## Budget law for the next phase

### Default

Codex solo, sequential, high-discipline.

### Allowed

Composer 2 lanes for volume work **after** Codex frames the task.

### Reserved

High-cost multi-model or Daimon/Audit passes only for:

- law changes
- architecture pivots
- release posture
- contradiction-heavy reviews
- high-risk integration decisions

### Avoid

- opening many premium reasoning chats just to preserve the feeling of a large field
- parallelizing by default when sequential routing is enough
- spending expensive context on tasks that are mostly clerical or mechanical

## Bottom line

ION should not retreat from what it learned today.
But it does need to change how it operates tomorrow.

The right next posture is:

- preserve the law
- reduce stale protocol teaching
- rebuild minimal templates
- route mostly through Codex sequentially
- spend auxiliary agent budget only where it clearly buys something real

That is the safest path to an IDE-native ION that still deserves to later become the
API-native production system.
