# Current-Phase Staffing And Semantic Identity Next Workload Plan

## Why this is next

Current-generation ratification is now explicit.

That means the next lawful frontier should come from the remaining non-blocking open
items preserved in the ratified branch state, not from pre-ratification completion
work.

- staffing law and semantic identity for active support roles and external chassis

## Findings

### 1. The open frontier is already explicit in the ratified bundle

The current acceptance bundle says the branch still has this unresolved frontier:

- final constitutional staffing law and final semantic identity law for every external chassis are not globally settled

So the next workload does not need to be invented. It needs to be packetized.

### 2. Current-phase mount law is real, but intentionally non-final

`ROLE_CHASSIS_MOUNT_PROTOCOL.md` makes the current posture lawful enough for operation,
but its non-goals explicitly preserve:

- no final constitutional staffing law
- no claim that every live role already has a full semantic identity record

So the branch has working mount law, not final staffing closure.

### 3. The semantic layer is only partially populated

The current semantic-identity registry contains:

- `CODEX.semantic.yaml`
- `NEMESIS.semantic.yaml`
- `VICE.semantic.yaml`
- `VIZIER.semantic.yaml`

It does not currently contain semantic identity records for:

- Mason
- Thoth
- Vestige
- browser ChatGPT as an external carrier

That does not automatically mean those records must be created now.
It does mean the branch should decide whether current-phase mount packets plus boots are
enough, or whether some of these roles now require stronger registry identity.

### 4. Rank and precedence law is narrower than the live support field

`RANK_AND_PRECEDENCE_PROTOCOL.md` maps first-pass rank classes for:

- Vizier
- Codex
- Vice
- Nemesis

It does not yet define personnel law or rank placement for every active support role.

That is compatible with current operation, but it leaves the support field partly in a
current-phase bridge posture.

## Bounded objective

Produce one evidence-backed current-phase staffing and semantic identity decision
package that answers:

1. which active support roles are already strong enough to remain boot-plus-mount carried,
2. which roles, if any, now require semantic identity records in `ION/03_registry/semantic_identities/`,
3. whether browser ChatGPT should remain `EXTERNAL_UNMOUNTED` by default under current branch law,
4. and which staffing / topology surfaces are active startup authority versus stale or witness-only support material.

## Non-goals

This workload should **not**:

- invent a total constitutional hierarchy for every historical role,
- silently promote browser ChatGPT into a mounted internal role,
- widen bridge packets into canonical packet law,
- change kernel behavior,
- or reopen already-ratified current-generation completion claims.

## Recommended carrier posture

Start now:

- `Vestige` for older-estate staffing / naming / topology archaeology
- `Thoth` for exact active-branch evidence extraction

Hold for now:

- `Mason` unless a later bounded mutation surface clearly falls into code/test scope
- browser ChatGPT unless Codex later requests one bounded external witness pass on the
  resulting proposal

## Working rule

The next workload is not "mint semantic files by reflex."

The next workload is:

- prove what the live branch already settles,
- prove what it still leaves provisional,
- then change registry or protocol surfaces only if the evidence says stronger closure is
  now required.
