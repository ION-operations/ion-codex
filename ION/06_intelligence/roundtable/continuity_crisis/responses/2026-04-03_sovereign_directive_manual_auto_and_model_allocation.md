# Sovereign Directive: Manual + Automatic Continuity and Model Allocation

## Source Intent

The Sovereign directs that the team treat the following as active design constraints:

1. **Manual continuity work must begin immediately and continue even while automation is being built.**
2. **Context packages, capsules, mini capsules, and the templates governing them must all be restored as active operational surfaces.**
3. **Manual and automatic continuity systems should run side by side until both are fully operational.**
4. **Only after side-by-side confidence is established should fully automatic runs be tested as a primary mode.**
5. **Model allocation and model switching per role/chat must be treated as an explicit design problem, not an ad hoc convenience.**

## Continuity Directive

The Sovereign's current view is that drift becomes much worse when MINI/CAPSULE/context-package discipline is absent, and much better when those systems are functioning and governed by template law.

Therefore:

- every meaningful action should again be treated as a template-governed continuity event
- context packages should not wait for the final automation layer to exist
- if automation is absent, agents should perform the continuity work manually to protocol
- manual continuity and automated continuity should be compared side by side rather than treated as mutually exclusive stages

## Model Allocation Directive

The Sovereign also directs that the team think seriously about:

- smart model allocation per agent and per task
- whether `max` / `extended thinking` modes should be defaulted or reserved
- whether lower tiers of the same model family preserve continuity better than cross-company switching
- how context integrity behaves when a role switches chassis or mode
- whether same-family switching (e.g. Claude→Claude tier changes) is safer than cross-company switching
- whether mode changes alone (`max` on/off) are enough for some tasks without full chassis swapping

## Questions Now Added To The Roundtable

1. Which continuity objects should be manually maintained immediately, even before automation exists?
2. What exact templates are required for:
   - private capsules
   - mini capsules / routing views
   - context packages
   - compiled projections
3. What does a safe side-by-side manual vs automated continuity validation loop look like?
4. What model allocation matrix should exist per role, per task class, and per phase?
5. What continuity risks are introduced by:
   - same-company model tier switching
   - cross-company chassis switching
   - `max` / `thinking` mode toggles within the same role

## Immediate Implication

The continuity roundtable should no longer discuss only whether the current state is broken.
It should now also design:

- the **manual continuity recovery protocol**
- the **manual+automatic side-by-side validation protocol**
- the **model allocation / chassis-switching protocol**

These are now linked problems, not separate ones.
