## 2. Continuity Law Candidate

### Proposed ratifiable law

**1. Source continuity is private.**  
Each active role owns its own continuity objects. No role writes another role's continuity.

**2. Shared root continuity surfaces are projections or temporary manual substitutes.**  
They may be useful, but they are not the primary continuity body of every role.

**3. Interchange happens through public channels only.**  
Inbox tasks, signals, public intelligence artifacts, and explicit handoff/review packets.

**4. Manual mode is real ION.**  
If automation is absent, continuity updates must be performed manually to protocol.

**5. Automation is shadow until proven.**  
It compares against manual source continuity before becoming authoritative.

**6. The chat-death test is mandatory.**  
If a role cannot start a fresh chat, read its lawful continuity, and resume correctly, the system is not continuity-safe.

**7. Continuity-sensitive release requires the full governance chain where applicable.**  
For covered artifacts: Primary → Daimon → Nemesis → Sovereign.

## 4. Recovery Conditions That Remain Active

Ratifying the law does **not** mean these are solved.

### R1 — Clone scaling remains blocked

No clone scaling until one end-to-end lawful cycle is demonstrated:

task arrival → private continuity update → signal emission → projection generation/reconciliation → successful fresh-chat restart from continuity

### R2 — Root-as-source language must be corrected in active surfaces

Any active document, boot, or role instruction that still treats shared root MINI/CAPSULE/STATUS as universal raw source continuity must be patched.

### R3 — Automation remains non-authoritative until shadow proof

No continuity compiler or projection builder becomes primary until it matches manual source continuity for at least one full clean work cycle.

### R4 — Continuity-sensitive release requires visible Daimon participation

For covered artifact classes, continuity-sensitive release should remain blocked if the Daimon lane has no visible artifact for Nemesis to audit against the consolidated set.

### R5 — No phase/runtime claims beyond current reality

No claim should imply that the unified `ION/` runtime or authoritative compiled context pipeline already exists until the relevant implementation stages are actually landed.

### R6 — Role matrix still needs to be made explicit

The system still needs a single visible role table stating:

- does this role have private MINI?
- does it have private CAPSULE?
- does the boot read private first?
- may it write projections?
- may it emit task files?

## 5. Immediate implementation sequence after ratification

1. Patch active boots to align with the law
2. Create private continuity roots for active roles
3. Land and verify the physical bus (`inbox`, task conventions, signal flow)
4. Patch the major templates with explicit manual continuity update obligations
5. Run one full lawful work cycle and record source/projection drift
6. Only then promote shadow automation or widen clone scaling
