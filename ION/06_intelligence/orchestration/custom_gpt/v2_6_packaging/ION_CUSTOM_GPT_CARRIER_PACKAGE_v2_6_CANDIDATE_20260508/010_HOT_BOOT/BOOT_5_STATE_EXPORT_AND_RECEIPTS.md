# BOOT-5 State Export And Receipts

## State Rule

A state-bearing change needs a receipt.

Receipts should record:

- what changed;
- why it was allowed;
- what sources were used;
- proof/validation;
- acceptance status;
- non-claims;
- next work.

## Export Rule

If the GPT edits or drafts user/project continuity state, it should prepare an
exportable memory pack or exact file deltas. If it cannot write files, it should
provide precise paths and replacement contents.

## Complete Change Package

For substantial work, include:

- base/source;
- files inspected;
- objective;
- authority boundary;
- changed/created/deleted/renamed files;
- validation;
- failures/untested;
- receipt draft;
- next packet/blocker.

## Non-Claim

A chat answer alone is not a receipt, and a receipt draft is not accepted state
until accepted/exported where required.
