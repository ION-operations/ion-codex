# Center status judgment

## Purpose

This judgment collapses the major root families into one explicit status frame so the recovery atlas can answer a basic question cleanly:

**What is a real center, what is a witness, what is a wrapper, and what should not be mistaken for a primary organism line?**

## Core judgment

The estate should currently be read through these center classes:

### 1. Active current line
- current extracted ION branch

This is the strongest **current governance-explicit repaired witness**.
It is the main place where current startup/routing/template-law recovery is happening.
It is **not yet** the strongest historical runtime or activation center by itself.

### 2. Strongest historical executable centers
- `ION-BUILD` = strongest historical runtime/API/session center
- `operation-victus` = strongest historical executable activation-authority center

These are the two most important missing executable centers relative to the current branch.

### 3. Strongest historical compact witnesses
- production `ION` = compact extracted-core witness
- production `ION (codex branch)` = sibling compact witness
- `IONv2` = compact executable implementation witness

These preserve compression that the current branch does not preserve in the same way.

### 4. Strong historical continuation witness
- `Project-Gemini`

This should be read as a strong rehomed executable continuation witness of the Victus manager/swarm line, not as merely a packaging echo and not as a wholly separate origin.

### 5. Strong historical constitutional/meta-template center
- older `ION-BUILD`/Aether template-development line

This remains constitutionally stronger than the current branch bridge-era template-evolution repair.

### 6. Mixed sibling witnesses
- `SOS`
- `SOS-Gemini`
- `SOS-OPUS`
- `AETHER-OS-V4`

These are not primary centers of the total organism, but they are too executable and too structurally rich to be flattened into doctrine residue.

### 7. Archaeology/atlas/support centers
- `00_CONSOLIDATED_ATLAS`
- `ProjectOpus`
- `ATLAS`

These should not be confused with executable organism centers, but they are crucial to understanding the project and recovering what kept being lost.

### 8. Wrapper and peripheral witnesses
- `composeraudit`
- `geminiaudit`
- `conjugate-basis-hidden-field`

These are real and valuable, but they should not silently inflate into primary centers.

## Practical rule

A future worker should not ask only:
- "what is newest?"
- "what runs?"
- "what is most documented?"

They must also ask:
- what kind of center is this?
- what does it preserve best?
- what does it fail to preserve?
- what complementary center is still missing if this one is taken as primary?

## Recovery consequence

A trustworthy reconstructed ION cannot be built from the current branch alone.
It must learn from at least:
- the current branch for repaired startup/template law
- `ION-BUILD` for runtime/API/session and meta-template constitutional strength
- `operation-victus`/`Project-Gemini` for executable activation/manager/swarm truth
- compact witnesses like precursor `ION` and `IONv2` for compression discipline
