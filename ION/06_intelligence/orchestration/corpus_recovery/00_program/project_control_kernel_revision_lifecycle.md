# Project control kernel revision lifecycle

## Purpose

This file gives the smallest practical lifecycle for revising the frozen default project control kernel without falling back into ad hoc drift.

## Lifecycle

1. **Detect a real insufficiency**
   - A control failure, repeated reassessment, ordering defect, defaults defect, or genuinely missing control primitive must be explicit.

2. **Check whether the kernel is actually the problem**
   - Confirm the issue is not better handled by: wider center consultation, conflict preservation, or a narrower work posture.

3. **Open a declared thaw path**
   - Use `project_control_kernel_revision_and_thaw_protocol.md`.

4. **Select the smallest lawful change class**
   - Use `project_control_kernel_change_classes.csv`.
   - Confirm approval expectations in `project_control_kernel_revision_approval_matrix.csv`.

5. **Create a revision record**
   - Use `project_control_kernel_revision_record_template.md`.

6. **Choose the result path**
   - clarification only
   - ordering correction
   - defaults correction
   - new control primitive
   - freeze rollback

7. **Re-freeze explicitly**
   - Any accepted change must end in a new freeze judgment or an explicit rollback to the prior frozen state.

## Core discipline

The revision lifecycle exists so the frozen kernel can evolve only through:
- explicit insufficiency evidence
- smallest-change discipline
- preserved conflict awareness
- auditable re-freeze behavior
