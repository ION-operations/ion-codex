# Project Control Kernel Governance Cadence

## Purpose

The frozen default project control kernel should not drift through neglect and should not thaw through convenience.
This cadence defines the smallest repeatable governance rhythm for reviewing the kernel without turning review into constant churn.

## Cadence layers

### Layer 1 — Event-driven review
Open a governance review immediately when any of the following occurs:
- repeated packet-gate failure on the same question class
- repeated widening that returns under the same insufficiency condition
- repeated reassessment triggered by the same mismatch
- preserved conflict that blocks ordinary work-start repeatedly
- evidence that the frozen kernel pushes future work toward false novelty or wrong-center consultation

### Layer 2 — Rolling health review
Review the kernel after a meaningful cluster of work-starts, not after every packet.
A rolling review should ask:
- are the defaults still routing questions to the right preserved centers?
- are question classes still discriminating enough?
- are answer/output classes still narrow enough?
- are landing boundaries preventing silent mutation?
- are return/reassessment rules stopping drift early enough?

### Layer 3 — Milestone review
When the project enters a materially different era of work, reopen the kernel review at the milestone level.
Examples:
- transition from recovery-heavy work to implementation-heavy work
- reintegration of a historically stronger center into the active branch
- successful restoration of a historical meta-template center
- replacement of a bridge-era surface by a stronger restored surface

## Governance default

The default stance is:
- **do not revise the kernel merely because a new pass feels more elegant**
- **do review the kernel when repeated work-start experience shows a stable insufficiency**

## Output expectation from governance review

A governance review should end in one of these outcomes:
- keep kernel frozen as-is
- clarification-only revision
- defaults correction
- new control primitive proposal
- freeze rollback consideration

Each of those outcomes must still pass through the thaw/revision protocol.


## Governance outcomes companion layer

After a governance review, classify the result through:
- `project_control_kernel_governance_outcome_classes.md`
- `project_control_kernel_governance_outcome_defaults.csv`

This helps keep review results small, auditable, and proportionate.
