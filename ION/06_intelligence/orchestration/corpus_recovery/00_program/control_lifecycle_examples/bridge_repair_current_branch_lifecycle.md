# Bridge-repair current-branch lifecycle

## Question
How should a narrowly scoped current-branch startup/routing clarification proceed when the atlas already shows the current branch is the primary active line for startup truth?

## Question class
`current_branch_bridge_repair`

## Primary center
current extracted branch

## Default posture
`bridge_repair`

## Answer/output class
`bridge_repair_note`

## Landing boundary
`current_branch_bridge_repair_only`

## Horizon framing
- current process position: current-branch clarification without widening beyond startup/routing layer
- near horizon: identify whether the issue is already covered by the current branch control surfaces
- mid horizon: if not, add a narrow clarification without changing preserved historical judgments
- far horizon: keep current-branch operational surfaces aligned with the atlas without local reinvention
- completion condition for this pass: a narrow repair note or doc change lands only in current-branch operating surfaces
- reassessment trigger: if the repair would require changing historical center judgments or widening into a new subsystem

Current-branch bridge-repair surfaces only.
