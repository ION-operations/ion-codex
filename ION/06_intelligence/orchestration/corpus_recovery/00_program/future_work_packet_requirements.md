# Future Work Packet Requirements

## Purpose

This file defines the minimum requirements for any future work packet that claims to extend, repair, compare, or reinterpret the ION/Aether/AIMOS estate after Milestone 1 atlas stabilization.

The main goal is to prevent a repeat of the historical failure mode:
- starting from the wrong surviving center,
- skipping the atlas,
- and inventing a local replacement for something that already exists elsewhere in the organism.

---

## Rule

No future work packet is valid unless it explicitly records how the working atlas base was consulted before the work began.

A future work packet that does not satisfy these requirements should be treated as:
- ungrounded,
- non-landable,
- and likely reinvention-prone.

---

## Required fields

Every future work packet must include all of the following.

### 1. Work question
- What is the exact question, burden, or repair being attempted?
- What class of question is it?
  - startup/routing
  - runtime/API/session
  - activation/manager/swarm
  - template/meta-template
  - compact implementation comparison
  - values/soul
  - archaeology/atlas
  - other

### 2. Atlas gate consultation
- Which entry surfaces were read first?
  - `milestone1_completion_judgment.md`
  - `atlas_base_usage_contract.md`
  - `atlas_work_gating_protocol.md`
  - `future_work_gating_authority_matrix.csv`
  - `working_center_selection_guide.md`
  - `conflict_register.md`
- What date/version of the atlas base was used?

### 3. Primary authority surface
- Which preserved center was selected first, and why?
- What exact file(s) were treated as the primary authority surface?
- What secondary centers were compared before acting?

### 4. Conflict check
- Which conflicts from `conflict_register.md` are relevant?
- Were any missing-evidence entries relevant?
- If the work begins despite unresolved conflict, why is that lawful?

### 5. Work posture
The packet must explicitly declare one of these work postures:
- consult only
- compare centers
- recover first
- bounded extension
- bridge repair

### 6. Non-reinvention claim
The packet must explicitly state one of the following:
- this work already exists elsewhere and is being recovered
- this work partially exists and is being reconciled
- this work does not appear to exist in the estate and is therefore a justified bounded extension

### 7. Expected artifact class
What kind of output is expected?
- comparison note
- recovery receipt
- system profile update
- executable receipt
- bounded code patch
- atlas update
- conflict entry
- other

### 8. Landing boundary
- What landing-boundary class applies?
- What can this work change?
- What can it not change?
- Does it change atlas interpretation only, or does it propose implementation mutation?

### 9. Orchestration foresight
- What is the current process position?
- What is the near horizon?
- What is the mid horizon?
- What is the far horizon?
- What counts as complete enough for this pass?
- What does this pass explicitly not complete?
- What would trigger reassessment?

---

## Minimum acceptance rule

A future work packet may only be treated as admissible if:
- the question class is explicit,
- the atlas gate consultation is explicit,
- the primary authority surface is explicit,
- the conflict posture is explicit,
- the work posture is explicit,
- and the non-reinvention claim is explicit.

If any of those are missing, the packet should be rejected or returned for repair.

---

## Why this matters

The estate repeatedly failed because work began from:
- memory,
- tool convenience,
- local branch familiarity,
- or a plausible but incomplete center.

These packet requirements exist to force future work to begin from the recovered atlas base instead.


## Additional required fields

Future work packets should also include:
- expected answer/output class
- landing-boundary class
- whether the output is non-widening, limited-widening, or blocked pending recovery
- the current process position and near/mid/far horizon
