## Why Lane A wins now

- the current branch already carries live template-governance bridge surfaces
  that are explicitly provisional and not final canon
- the current-phase template registry names active bridge surfaces and
  `PROVISIONAL_BRIDGE_NOT_FINAL_CANON` surfaces directly, which means the gap is
  real and current-line visible rather than purely archaeological
- the Era 2 lane matrix already marks Lane A as a `recover_first` lane, which
  is narrower than fresh implementation and consistent with the project control
  kernel
- the authorized branching topology already preserves a bounded Lane A
  background review lane, so this selection does not invent a new work family

## Why Lane D is not selected now

- the compactness conflict is real, but the current evidence surface is still
  primarily a conflict note rather than a lane-ready recovery stack
- Lane D would begin as `compare_centers`, but the current active branch does
  not yet have the same level of present-tense bridge pressure that Lane A now
  has through template-governance surfaces
- selecting Lane D now would risk turning an important value conflict into a
  vague style debate before the template/meta-governance center is recovered
  strongly enough

## Why explicit hold is not selected now

- a no-new-lane hold would be lawful if evidence were still too weak to choose
- current evidence is strong enough to name one bounded next lane without
  implementation widening
- leaving the board in a placeholder hold would recreate the same recursion
  this packet exists to close

## Selection law

- one bounded Lane A recover-first packet

- active-law emission for Lane A by implication
- fresh runtime/session widening
- compactness reconciliation by side-channel
- broader template canon closure without a bounded recovery packet first
