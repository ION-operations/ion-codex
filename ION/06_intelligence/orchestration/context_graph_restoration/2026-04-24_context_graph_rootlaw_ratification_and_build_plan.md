# Context Graph Root-Law Ratification and Build Plan

## 1. Ratification claim

The recovered root law is not new doctrine. It is the original body-plan of ION returning to explicit current-phase visibility:

> ION is a living context graph operated by lawful agents through templates.

This packet should be read as a restoration and ratification candidate. It does not replace existing kernel law, role law, packet law, schedule law, or template law. It gives those organs their missing substrate interpretation.

## 2. Why this matters

Without the context-graph root law, ION appears as a collection of strong but partially disconnected mechanisms: roles, packets, templates, daemon receipts, bounded parallelism, context modes, semantic registries, and continuity bundles.

With the root law restored, those mechanisms become one system:

- files, packets, receipts, templates, protocols, semantic identities, tests, signals, and work units are context nodes;
- dependencies, provenance, authority, contradiction, ownership, settlement, and supersession are context edges;
- bounded context is a graph-neighborhood compilation, not a paste operation;
- agents are graph-region managers and custodians;
- subagents are specialist custodians over bounded graph regions;
- fan-out is lawful graph-region delegation;
- fan-in is lawful settlement back into parent graph continuity;
- templates are graph-operation law;
- receipts are graph-mutation and settlement evidence;
- zips and bundles are portable continuity-graph capsules.

## 3. Recovery evidence spine

The Apr 19 lineage contains direct evidence that ION began as graph substrate architecture rather than a generic workflow system.

### 3.1 IONv2 model law

The early model layer defined each ion as simultaneously file, program, agent, specification, memory, and documentation. This is precisely the ancestor of current context nodes: one durable object that can carry meaning, execution, specification, memory, and documentation together.

### 3.2 IONv2 graph law

The early graph layer explicitly described the ION network as a directed graph where ions are nodes and bonds are edges. It included dependency chains, impact analysis, topological ordering, cycle detection, and connected-component analysis.

### 3.3 IONv2 context compiler law

The early context compiler assembled LLM context by walking the ion graph. This is the ancestor of current bounded context compilation and directly contradicts any interpretation of ION as naive RAG or paste-history.

### 3.4 Sub-capsule spatial context law

The sub-capsule audit stated that complexity should be managed by spawning distinct nodes in the cognitive graph, not by stuffing all nuance into the root agent capsule. This is the ancestor of manager-to-subspecialist fan-out and graph-region ownership.

### 3.5 Current ION 3 continuity

Current ION 3 already contains the living descendants: bounded parallelism, child work, fan-in settlement, context modes, context planes, role mounts, single-carrier traversal, manual/automation equivalence, and continuation bundles. The missing piece was the explicit substrate declaration binding them together.

## 4. Build phases

### Phase A — Restore substrate law

Add and ratify:

- `CONTEXT_GRAPH_SUBSTRATE_PROTOCOL.md`
- `CONTEXT_NODE_AND_PACKAGE_PROTOCOL.md`
- `AGENT_GRAPH_CUSTODIAN_AND_SUBSPECIALIST_FANOUT_PROTOCOL.md`
- `context_graph_substrate_registry.yaml`
- core context templates

Success condition: the current root can explain ION from the one-line root law and map every major subsystem to nodes, edges, packages, agents, templates, receipts, or graph projections.

### Phase B — Formalize graph node and edge schemas

Create the first machine-readable schemas for:

- context node envelope;
- context edge envelope;
- context package envelope;
- graph-region jurisdiction;
- fan-out request;
- fan-in settlement;
- graph writeback receipt.

Success condition: new work artifacts can declare themselves as graph nodes and can be related to existing nodes by typed edges.

### Phase C — Register agent graph jurisdictions

Promote the manager/subspecialist lattice from prose into registry:

- Steward: lawful continuation and current-phase routing graph;
- Vizier: architecture and cross-region synthesis graph;
- Vice: contradiction and future-answerability graph;
- Nemesis: evidence, audit, tests, release-risk graph;
- Mason: implementation and runtime graph;
- Scribe: documentation, archive, human-legibility graph;
- Vestige: donor-lineage and lost-surface graph;
- Relay: intent, packet, signal, external-return graph;
- Thoth: research and evidence-extraction graph;
- Atlas: topology, comparative map, dependency graph;
- Weaver: UI/projection graph when active.

Success condition: every persistent graph region has a primary manager and audit relation.

### Phase D — Bind templates to graph operations

Define every major template family by graph operation:

- node creation;
- node mutation;
- edge creation;
- package assembly;
- fan-out request;
- fan-in settlement;
- contradiction registration;
- receipt emission;
- projection/membrane rendering;
- portable capsule export.

Success condition: templates are no longer just document shapes; they are governed graph-operation forms.

### Phase E — Implement minimal kernel graph substrate

Only after law and registries are settled, implement or restore runtime support:

- context node store;
- context edge store;
- graph traversal/package assembler;
- region ownership lookup;
- fan-out package constructor;
- fan-in settlement recorder;
- writeback receipt emitter.

Success condition: a manager can request a graph region, spawn sub-specialists with bounded context packages, and settle their outputs back into graph continuity.

## 5. Anti-drift tests to add

ION should not be allowed to forget the root law again. Add tests or validation surfaces checking:

1. `CONTEXT_GRAPH_SUBSTRATE_PROTOCOL.md` exists and contains the root law.
2. `context_graph_substrate_registry.yaml` declares node types, edge types, and manager jurisdictions.
3. Every active agent roster entry has either graph jurisdiction or explicit exclusion.
4. Context package templates exist and contain source node / edge / package fields.
5. Fan-out and fan-in templates exist and reference manager, subagent, node set, settlement output, and receipt fields.
6. No current-phase doc defines ION primarily as a flat prompt, flat file tree, or ordinary RAG system without referencing the context graph substrate.
