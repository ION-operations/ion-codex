# DOCTRINE EVOLUTION PROTOCOL

**Status:** Current-phase operational proposal / candidate governance protocol  
**Date:** 2026-04-24  
**Authority posture:** A3 until reviewed and ratified  
**Purpose:** Define how ION may evolve its own doctrine, constitution, protocols, templates, registries, role canon, root invariants, and explanatory surfaces without silently converting insight into law.

---

## 1. Controlling law

```text
ION may evolve its own doctrine only through the same living-context-graph,
template, authority, receipt, contradiction, and governed-write law that it
imposes on every other action.
```

Short form:

```text
ION must modify ION law lawfully.
```

This protocol exists because ION is now operating on itself. Self-modification is a privileged risk class. It can repair lost law, but it can also launder temporary insight into canon, overwrite donor truth, or let excitement impersonate ratification.

---

## 2. What counts as doctrine evolution

Doctrine evolution includes any action that changes, adds, retires, reinterprets, promotes, or materially reframes:

```text
constitutional law
root invariants
governance protocols
template law
semantic identity / true-name surfaces
role canon or agent jurisdiction
approved context rules
graph substrate rules
evented file graph behavior
runtime proof claims
public product explanation
builder implementation order
```

Ordinary typo correction is not doctrine evolution unless it changes meaning.

---

## 3. Doctrine modification depth classes

Every doctrine change must declare a depth class.

```text
D0 — wording / typo / formatting only
D1 — terminology clarification without semantic change
D2 — protocol, template, or documentation addition
D3 — registry, role-jurisdiction, template-behavior, or approved-context change
D4 — constitutional interpretation change
D5 — root invariant, self-definition, or organism-wide doctrine change
```

### Required review by depth

```text
D0: Scribe or responsible owner
D1: Scribe + owner
D2: Steward + owner + Nemesis spot-check
D3: Steward + owner + Nemesis + affected agent family
D4: Steward + Vizier + Vice + Nemesis + Scribe
D5: Steward + Vizier + Vice + Nemesis + Scribe + Vestige + Atlas, with donor-line comparison
```

A D5 change may not self-ratify.

---

## 4. Mandatory doctrine-evolution pipeline

Every D2-D5 change must pass the following pipeline:

```text
1. Insight capture
2. Classification
3. Authority posture declaration
4. Donor-line search
5. Existing canon comparison
6. Contradiction review
7. Re-expression validation
8. Runtime / implementation mapping
9. Formal surface draft
10. Proper-organ review
11. LAND / HOLD / ESCALATE / REJECT verdict
12. Registry update proposal
13. Test guard
14. Receipt / changelog / handoff
```

D0-D1 changes may use a lighter version but must not hide semantic impact.

---

## 5. Pipeline details

### 5.1 Insight capture

A major realization begins as an insight, not as law.

Required fields:

```yaml
insight:
  statement:
  captured_by:
  captured_at:
  suspected_depth:
  affected_regions: []
  proposed_status:
  why_it_matters:
  immediate_risks: []
```

### 5.2 Classification

Classify the change as one or more of:

```text
root_invariant_restoration
constitutional_amendment
doctrine_clarification
protocol_addition
template_evolution
role_evolution
runtime_implementation
registry_update
explanatory_document_update
product_claim_update
```

### 5.3 Authority posture declaration

Every draft must say whether it is:

```text
A1_CROWN_LAW
A2_PRODUCTION_GOVERNANCE
A3_OPERATIONAL_PROPOSAL
PROVISIONAL_RESTORED_ROOT_INVARIANT
PROVISIONAL_MECHANICAL_LAW
RUNTIME_PROOF_SURFACE
WITNESS
HISTORICAL_LINEAGE
EXPLANATORY_SYNTHESIS
```

No doctrine file may leave authority posture implicit.

### 5.4 Donor-line search

Ask whether this idea already existed under another name in:

```text
AIMOS
Aether
Aletheion
IONv2
ION Apr 19 line
current ION root
forensic restoration packages
prior encyclopedias
prior runtime missions
```

The search result must distinguish:

```text
direct evidence
partial ancestor
similar but not same
contradictory ancestor
no evidence found
```

### 5.5 Existing canon comparison

Compare the proposed doctrine against:

```text
current constitution / 11 laws
current architecture protocols
semantic identity registry
template registry
role roster / boot surfaces
kernel modules
runtime tests
approved context surfaces
```

### 5.6 Contradiction review

The review must list:

```text
what this contradicts
what this supersedes
what this reinterprets
what older names become stale
what runtime surfaces are not yet aligned
what remains unresolved
```

Contradictions are preserved unless explicitly settled.

### 5.7 Re-expression validation

The doctrine must be expressible at all required levels without semantic collapse:

```text
one-sentence explanation
product explanation
builder explanation
runtime explanation
constitutional explanation
technical/schema explanation
testable invariant
```

### 5.8 Runtime / implementation mapping

The doctrine must map to:

```text
files
protocols
registries
templates
kernel modules
tests
receipts
runtime evidence
remaining gaps
```

If no runtime enactment exists, the doctrine must say so.

### 5.9 Formal surface draft

The doctrine may become:

```text
protocol
semantic identity
registry entry
template
test
task packet
amendment packet
runtime implementation plan
explanatory document update
```

### 5.10 Proper-organ review

Required organs depend on class and depth.

Default mapping:

```text
Steward = routing and current-phase approval posture
Vizier = architecture/coherence synthesis
Vice = contradiction/future-risk pressure
Nemesis = independent audit
Scribe = documentation/canon expression
Vestige = donor-line recovery
Atlas = topology/mapping
Mason = implementation feasibility
Thoth = theory/research relation where needed
```

### 5.11 Verdict

Allowed verdicts:

```text
LAND_RATIFIED
LAND_PROVISIONAL
LAND_EXPLANATORY_ONLY
HOLD_PENDING_DONOR_COMPARISON
HOLD_PENDING_RUNTIME_PROOF
HOLD_PENDING_CONTRADICTION_REVIEW
ESCALATE_AUTHORITY_CONFLICT
ESCALATE_SEMANTIC_CONFLICT
REJECT_DUPLICATE
REJECT_FALSE_CANON
REJECT_PREMATURE
```

### 5.12 Registry update proposal

Registry writes are proposals until governed write / review conditions are met.

### 5.13 Test guard

Every D3-D5 change requires a test guard or a written reason why no test can yet exist.

### 5.14 Receipt / changelog / handoff

Every completed doctrine evolution pass must emit residue:

```text
doctrine_evolution_receipt
canon_promotion_receipt
contradiction_review_receipt
re_expression_validation_receipt
runtime_mapping_receipt
changelog entry
handoff / next action
```

## 6. Evented-file integration

A completed doctrine evolution packet is a template-instantiated graph object. When complete, it may trigger:

```text
semantic review
registry update proposal
constitutional amendment review
Nemesis audit
runtime mapping request
test creation request
approved-context index update
```

It may not trigger direct canon promotion without the required verdict.

---

## 7. Non-loss clauses

This protocol is invalid if interpreted to allow:

1. excitement to impersonate ratification;
2. generated prose to become law without review;
3. donor-line evidence to be skipped;
4. contradictions to be smoothed away;
5. old law to be overwritten silently;
6. runtime proof claims without implementation mapping;
7. registry mutation without receipt;
8. self-certification by a single agent on D4-D5 changes;
9. “new feature” framing for recovered origin law without lineage review;
10. doctrine changes without approved-context tagging.
