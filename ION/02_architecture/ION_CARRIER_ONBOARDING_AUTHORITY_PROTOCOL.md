# ION Carrier Onboarding Authority Protocol

## Purpose

ION needs carrier onboarding that does not depend on shell-root markdown or
let any host-specific adapter become ION law.

Carrier onboarding authority lives in registry profiles, role boot files, active runtime packets, and carrier execution templates.

The former root files `AGENTS.md` and `START_HERE_FOR_ANY_AGENT.md` are retired
from hot root authority as of V123. If they appear at shell root with procedural
onboarding content, that is a drift finding. Their historical contents belong
in containment or archive evidence, not active carrier mount order.

The April 17 root-authority bundle that centered `START_HERE.md` is also
retired from hot startup authority. It is containment evidence, not a first-read
carrier mount surface.

## Authority Order

1. Confirmed shell root containing `pyproject.toml` and `ION/REPO_AUTHORITY.md`
2. `ION/REPO_AUTHORITY.md`
3. Active runtime packets under `ION/05_context/current/`
4. Current branch locks and consolidation reports
5. Kernel audits and generated receipts
6. Carrier-specific profile/protocol/template files
7. Host shortcuts, saved agent definitions, contained shell-root markdown, MINI/CAPSULE, and chat memory as witness context only

## Carrier Onboarding Substrate

The enforceable onboarding substrate is:

```text
ION/03_registry/*carrier_profile.yaml
ION/03_registry/*runtime_identity_mount_registry.yaml
ION/03_registry/boots/*.boot.md
ION/07_templates/carriers/*.md

ION/05_context/current/ACTIVE_CARRIER_TURN_PACKET.json
ION/05_context/current/ACTIVE_ROLE_SPAWN_PLAN.json
```

Shell-root markdown must not be used as the carrier onboarding substrate.

## Root Shim Retirement Law

Root onboarding shims are retired from hot authority.

They must not contain procedural command lists, authority anchors, mandatory
reads, carrier id routing instructions, or protocol summaries. Those belong in
ION-native protocols, registries, execution packet templates, active packets,
and kernel status/audit surfaces.

## Adapter-Specific Law

Cursor-specific commands belong in Cursor adapter surfaces:

```text
.cursor/commands/
.cursor/rules/
ION/02_architecture/ION_CURSOR_CANONICAL_WORKFLOW_UNIFICATION_PROTOCOL.md
```

Codex-specific carrier limits belong in:

```text

ION/03_registry/codex_extension_carrier_profile.yaml
ION/07_templates/carriers/CODEX_EXTENSION_EXECUTION_PACKET.md
```

MCP-specific carrier limits belong in MCP bridge, MCP donor, and MCP mount-session surfaces.

## Forbidden Root-Onboarding Drift

If root compatibility files exist, they must not say or imply:

```text
START HERE - ION Cursor Carrier, V94
AGENTS.md - ION Cursor Carrier Canonical Workflow, V94
Run the V94 CLI spine
You start as CURSOR_CARRIER_CONTROL_SURFACE
/ion is the universal primary ION command
ion_cursor_autopilot_packet is the universal carrier spine
Current Protocol Anchors
Mandatory reads
kernel.ion_carrier_continue
kernel.ion_status
ION_CURRENT_OPERATING_PACKET
```

Those statements may remain in historical Cursor-specific protocols where scoped correctly, but they are stale when presented as root-level onboarding for every carrier.

## Preservation

`AGENTS.md` and `START_HERE_FOR_ANY_AGENT.md` are protected from silent loss by
the containment policy because prior AI packaging dropped project organs.
Protection does not freeze them in hot authority and does not make them primary
carrier authority. V123 retires their hot-root copies through containment with
matching SHA-256 proof.

V123 also retires
`ION/05_context/exports/2026-04-17_root_authority_bundle/` to containment
because it carried `START_HERE.md` as a first-read startup surface.

## Exit Condition

Carrier onboarding authority is healthy when:

- shell root confirmation remains the first invariant
- registry/profile/template onboarding surfaces are present
- root compatibility files are absent from hot root or are non-procedural tombstones only
- Cursor is represented as one adapter, not the organizing doctrine
- Codex has its own carrier id and profile
- no-silent-loss packaging protects root onboarding changes without blocking lawful containment movement
- tests or audits detect stale or procedural root onboarding language
