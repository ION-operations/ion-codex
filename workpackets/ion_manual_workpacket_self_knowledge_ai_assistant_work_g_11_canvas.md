# 1. Executive summary

We mounted ION and advanced two connected worklines.

First, we activated a candidate **ION Self-Knowledge System** so future carriers can answer ION-about-ION questions by mounting a routed domain system instead of reconstructing ION from scattered files, summaries, memories, and guesses.

Second, we discovered and began correcting a larger missing layer: **AI assistant / IDE work itself** had not yet been fully defined as a governed domain system. ION had meta-templates, dynamic domain expansion machinery, and some skill-like templates, but the actual world of chat/IDE/CLI/PR/background-agent work needed to be domainized.

The key correction:

```text
Do not route specialist work to a generic coder or writer first.

UI work is a specialist domain.
Documentation work is a specialist domain.
PR review is a specialist domain.
Terminal proof is a specialist domain.
Release evidence is a specialist domain.
Migration work is a specialist domain.
Background queue/result intake is a specialist domain.
IDE work itself is a specialist domain.
```

This work produced candidate domains, agents, templates, protocols, state families, dataset entries, fission expansions, boot drafts, machine-checkable template specs, valid/rejection template examples, boot exercises, and a route compiler.

Nothing has been accepted into permanent ION law yet.

# 16. What this means for ION conceptually

## 16.1 Self-knowledge correction

ION should not be rediscovered by each carrier.

It should be mounted.

Correct behavior:

```text
ION-about-ION request
→ mount self-knowledge packet
→ classify query
→ load route
→ load authority-ranked domains
→ answer/action with non-claims
```

## 16.2 Assistant-work correction

AI assistant work should not be treated as generic “coding with AI.”

It is a large governed workfield with multiple domains.

Correct behavior:

```text
assistant-work request
→ classify embodiment and intent
→ choose specialist domain
→ mount specialist agent
→ instantiate governing template
→ collect proof
→ settle result
→ receipt/update continuity
```

## 16.3 Agent correction

An agent is not just:

```text
model + role prompt + tools
```

In ION, an agent is better understood as:

```text
domain-local inference under governance
```

or:

```text
LLM carrier + governed domain + context package + template + proof obligation + settlement route
```

This is why specialist agents matter.
