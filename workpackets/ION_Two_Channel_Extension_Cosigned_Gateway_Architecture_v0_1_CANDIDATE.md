# ION Two-Channel Extension-Co-Signed Gateway Architecture — v0.1 Candidate

**Date:** 2026-05-08  
**Status:** CANDIDATE / not accepted state  
**Purpose:** Perfect the multi-user routing model where one ION Custom GPT action surface can serve many users while private workspace data is released only when the user’s authenticated browser extension co-approves the exact request.

---

## 0. Executive compression

ION needs a **two-channel authorization gate**.

```text
GPT Action channel:
  The ION Custom GPT asks the gateway for data or work.

Extension proof channel:
  The user’s signed-in browser extension approves the exact pending request.

Gateway decision:
  The gateway releases scoped data only when both channels match.
```

The GPT may request.  
The extension may prove.  
The gateway may decide.  
The receipt may make the result inheritable.

---

## 1. Design goal

Any user should be able to open the ION Custom GPT and use the same public conversation starters:

```text
/sign-in
/sign-up
/guest-mode
/what is ION?
```

After sign-in, the GPT can use one global action endpoint, but the gateway must route each request to the correct user/workspace/sandbox without leaking one user’s data to another.

The system must prevent this failure:

```text
Attacker opens same public GPT
→ types another user_id or stolen connection_ref
→ asks for that user’s VM/workspace data
→ gateway returns private data
```

Required behavior:

```text
Attacker opens same public GPT
→ creates request with another user’s handle/ref
→ gateway creates or rejects pending request
→ attacker lacks the signed-in extension/browser proof
→ gateway releases no private data
```

---

## 2. Non-negotiable laws

1. **No passwords in GPT chat.**
2. **No OAuth tokens, API keys, cookies, SSH keys, or device private keys in GPT chat.**
3. **A user handle, user_id, connection_ref, or workspace_ref is not authority.**
4. **A GPT Action request is a candidate action, not proof.**
5. **Private data release requires extension co-approval or an explicitly allowed low-risk policy.**
6. **All sensitive actions need scoped request IDs, nonces, expiry, one-time use, and receipts.**
7. **Codex/CLI execution remains approval-gated; first demo writes bounded work packets only.**

## 3. Actors and trust boundaries

### 3.1 ION Custom GPT

Role:

```text
front door
conversation carrier
request drafter
context/package explainer
state/proof reader
action caller
receipt consumer
```

Allowed to hold:

```text
connection_ref
workspace_ref
request_id
receipt_id
state_root_hash
allowed scopes
sanitized state summaries
```

Forbidden to hold:

```text
password
OAuth access token
refresh token
session cookie
API key
device private key
local filesystem secret
Codex execution authority
```

### 3.2 Browser extension

Role:

```text
user-facing auth panel
same-browser/chat pairing surface
device key holder
request approval UI
local gateway/hosted gateway bridge
ion_reentry proof injector
```

The extension is part of the trusted client surface for this architecture. It does not make final authorization decisions by itself; it supplies user/browser/device proof to the gateway.

### 3.3 ION Gateway

Role:

```text
identity authority
workspace router
request state machine
scope checker
challenge issuer
signature verifier
receipt ledger
VM/local-edge router
```

The gateway is the central decision point.

### 3.4 Local Edge Node / VM / Sandbox

Role:

```text
per-user execution/data cell
workspace storage
Codex work-packet writer
optional future Codex runner
local filesystem adapter
```

For the demo, Braden’s PC can act as the first VM cell. Later, this becomes a hosted VM/container or user-local edge node reached through outbound connection.

### 3.5 Codex CLI

Role:

```text
implementation worker
local filesystem/test carrier
```

Codex must receive bounded work packets. It must not be invoked as a raw unrestricted shell by the GPT.

## 4. Credential classes

### 4.1 User account credential

Used only outside GPT chat.

```text
password/passkey/OAuth/magic link
entered into extension or gateway UI
stored/verified by gateway or identity provider
never copied into GPT
```

### 4.2 Device key pair

Generated by extension/local node.

```text
private key:
  stays in extension storage / local secure storage

public key:
  registered with gateway for user/device

purpose:
  sign per-request gateway challenges
```

For browser demo, prefer WebCrypto ECDSA P-256 or Ed25519 if available. Use non-extractable keys when practical. If non-extractable persistence is too slow for the MVP, record that as a security downgrade and use extractable demo keys only in dev mode.

### 4.3 GPT Action credential

Configured in the Custom GPT Action layer.

```text
purpose:
  prove the API call came through the configured GPT Action surface

not purpose:
  prove which user owns the requested data
```

For local dogfood this may be a dev header key. Production should use OAuth/OIDC and server-side scopes.

### 4.4 Routing handles

Visible to the GPT, but not secrets.

```text
connection_ref
workspace_ref
state_root_hash
request_id
receipt_id
pairing_ref
chat_pairing_ref
```

These are identifiers and proof pointers. They do not grant access by themselves.

## 6. Protocols

## 6.1 Sign-up

```text
1. User clicks /sign-up in GPT.
2. GPT emits ion_connect sign_up request.
3. Extension detects request and opens sign-up UI.
4. User creates account outside chat.
5. Extension generates device key pair.
6. Gateway stores user + device public key.
7. Gateway creates workspace_ref and connection_ref.
8. Extension injects ion_reentry proof into GPT chat.
9. GPT mounts proof and continues.
```

No password or token enters the chat.

## 6.2 Sign-in

```text
1. User clicks /sign-in.
2. GPT emits ion_connect sign_in request.
3. Gateway creates pairing_ref or extension starts local pairing.
4. Extension opens sign-in UI.
5. User authenticates outside chat.
6. Extension proves device/user session to gateway.
7. Gateway binds connection_ref to user/workspace/device/chat_pairing_ref.
8. Extension injects ion_reentry proof.
```

## 6.3 Guest mode

```text
1. User clicks /guest-mode.
2. GPT emits ion_connect guest_mode request.
3. Gateway creates ephemeral guest workspace.
4. Extension may create local guest identity or gateway may create server guest identity.
5. GPT receives limited ion_reentry proof.
```

Guest scopes:

```text
allowed:
  orientation
  docs/explainer
  demo packets
  dry-run action proposals

forbidden:
  private data read
  hosted VM access
  Codex live execution
  persistent cross-session secrets
```

## 6.4 Sensitive data request

```text
1. User asks GPT: "Load my project state."
2. GPT calls gateway: POST /v1/requests
3. Gateway authenticates GPT Action channel.
4. Gateway validates connection_ref exists and scope is possible.
5. Gateway creates request_id + challenge_nonce.
6. Gateway returns PENDING_EXTENSION_APPROVAL.
7. Extension sees request_id through:
   a. gateway polling for this connection_ref, and/or
   b. fenced YAML/panel in the ChatGPT page.
8. Extension shows user: requested resource, scope, reason, expiry.
9. User approves.
10. Extension signs canonical request challenge.
11. Gateway verifies:
    - extension session user
    - device key
    - signature
    - request_id
    - connection_ref
    - chat_pairing_ref
    - scope
    - expiry
    - one-time nonce
12. Gateway releases sanitized result to GPT.
13. Gateway writes receipt.
```

## 6.5 Codex work-packet request

```text
1. GPT proposes a Codex packet.
2. Gateway creates pending request requiring codex.workpacket.write.
3. Extension shows exact target path/objective/boundary.
4. User approves.
5. Gateway writes the packet into the workspace inbox.
6. Gateway returns receipt.
```

Default demo mode writes packets only. It does not auto-run Codex CLI.

## 7. Same-browser/chat binding

The phrase “same browser” should be implemented as a **chat pairing binding**, not guessed from IP or user-agent.

Minimum demo binding:

```text
extension_instance_id
+ device_id
+ chat_pairing_ref
+ connection_ref
```

Stronger binding:

```text
extension generates chat_pairing_secret
gateway stores hash(chat_pairing_secret)
extension signs request challenges that include chat_pairing_ref
extension only approves requests it sees for that active ChatGPT tab/session
```

The gateway should not rely on raw browser fingerprints. It should rely on explicit registered device keys and pairing refs.

## 8. Request state machine

```text
NEW
→ PENDING_EXTENSION_APPROVAL
→ APPROVED
→ RELEASED
→ RECEIPTED
```

Failure exits:

```text
DENIED_BY_USER
EXPIRED
REJECTED_SCOPE
REJECTED_SIGNATURE
REJECTED_USER_MISMATCH
REJECTED_DEVICE_REVOKED
REJECTED_CHAT_PAIRING_MISMATCH
REJECTED_REPLAY
REJECTED_POLICY
```

A request can release data only once.

## 10. Data release rules

### 10.1 Data classes

```text
PUBLIC_ORIENTATION:
  What is ION, public docs, starter help.

GUEST_SAFE:
  demo docs, sample packets, empty workspace state.

PRIVATE_SUMMARY:
  current state summary, manifests, receipts.

PRIVATE_ARTIFACT:
  actual user files, packages, logs, source code.

MUTATION:
  writing work packets, updating state, queue changes.

EXECUTION:
  running Codex, shell commands, tests, external calls.
```

### 10.2 Required approvals

```text
PUBLIC_ORIENTATION:
  no extension approval needed

GUEST_SAFE:
  guest connection proof

PRIVATE_SUMMARY:
  extension co-approval or remembered consent policy

PRIVATE_ARTIFACT:
  extension co-approval every time in MVP

MUTATION:
  extension co-approval + receipt

EXECUTION:
  explicit user approval + proof gate + receipt; blocked in MVP
```

## 11. Threat model and controls

| Threat | Control |
|---|---|
| User handle spoofing | user_id/handle not authority; require extension-authenticated connection |
| Stolen connection_ref | short expiry + request-specific extension signature required |
| Cross-user data leak | server-side ACL on every request |
| Replay of approval | nonce, request_id, expiry, one-time release |
| GPT Action service key compromise | extension co-approval required for private/mutating data |
| Malicious prompt asks for token | GPT instructions forbid secrets; gateway never returns tokens |
| User B logged into same GPT | connection is bound to user/device/chat pairing |
| Extension wrong account | gateway checks user_id/device_id/workspace ACL |
| Stale request approval | expiry and canonical request hash |
| Path traversal | workspace path allowlist, normalized paths, no arbitrary paths |
| Codex overreach | packet-only MVP; explicit approval before execution |
| Guest escalation | guest workspace has separate ACL and blocked scopes |
| VM breakout | future hosted VMs require container isolation and resource limits |
| Extension supply-chain risk | signed releases, version receipt, manual install checks in demo |
| Data in MCP tool definitions | avoid sensitive data in tool JSON/tool descriptions |

## 13. MVP endpoint split

### GPT Action API

Safe for Custom GPT Action schema:

```text
GET  /health
POST /v1/pairings/start
GET  /v1/connections/{connection_ref}/status
POST /v1/requests
GET  /v1/requests/{request_id}/status
GET  /v1/requests/{request_id}/result
GET  /v1/receipts/{receipt_id}
```

No password endpoints here.

### Extension/Gateway API

Used by extension/local UI only:

```text
POST /v1/auth/signup
POST /v1/auth/signin
POST /v1/auth/logout
POST /v1/devices/register
GET  /v1/extension/pending
POST /v1/extension/requests/{request_id}/approve
POST /v1/extension/requests/{request_id}/deny
POST /v1/guest/start
```

These must not be exposed as GPT Actions.

## 14. Acceptance tests

### 14.1 Happy path

```text
sign up through extension
connection_ref injected into GPT
GPT requests workspace state
extension approves request
gateway releases sanitized state
receipt exists
```

### 14.2 Anti-spoof

```text
User B copies User A connection_ref
User B asks GPT for User A state
gateway creates pending or rejects
User B extension cannot approve as User A
no data released
receipt records denial
```

### 14.3 Stolen request replay

```text
approve request once
replay same approval signature
gateway rejects replay
```

### 14.4 Expired request

```text
create request
wait past expires_at
extension approval rejected
```

### 14.5 Wrong device

```text
same user but unregistered/revoked device signs
gateway rejects
```

### 14.6 Guest isolation

```text
guest workspace can read demo data
guest cannot list private workspaces
guest cannot create Codex execution request
```

### 14.7 Codex packet write

```text
GPT proposes packet
extension approves
gateway writes packet to inbox
receipt references path and hash
Codex is not executed automatically
```
