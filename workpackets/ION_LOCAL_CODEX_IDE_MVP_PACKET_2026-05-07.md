## Product target

Build a local IDE/cockpit that feels like a Codex IDE, but is governed by ION:

```text
ChatGPT Browser / Sev
-> browser extension cockpit
-> localhost ChatOps daemon
-> ION packets, receipts, queues, context packs
-> Codex CLI bounded worker invocation
-> proof-gated task return
-> Steward integration queue
-> receipt / next context
```

The IDE is not a second ION and not a second agent system. It is a front door/cockpit over the existing ION owners.

## MVP lanes

### Lane A — Local cockpit

Surface current status without mutating source:

- ION status
- active work packet
- active role spawn plan
- Codex work queue
- agent queue/status
- recent receipts
- daemon health
- blocked gates
- attachable artifacts

Owner surfaces already present:

- `ION/09_integrations/browser_extension/ion_chatops_bridge/`
- `ION/09_integrations/local_daemon/ion_chatops_bridge/`
- `ION/04_packages/kernel/ion_chatops_bridge.py`
- `ION/04_packages/kernel/ion_codex_queue_runner.py`
- `ION/04_packages/kernel/ion_agent_invocation_broker.py`

### Lane B — Custom GPT Action / public HTTPS bridge

Use Custom GPT Actions only as a bounded API front door. The action should call a public HTTPS tunnel or deployed bridge that forwards to the local daemon. Do not point the GPT action at raw localhost unless the platform can actually reach it.

Initial action scope:

- read health
- read policy
- validate action packet
- submit only approval-scoped bounded actions
- read Codex queue
- create Codex work packet

Blocked through action schema/policy:

- arbitrary shell
- direct delete
- git push
- credential access
- production deploy
- broad filesystem writes

### Lane C — Codex CLI worker loop

One packet should produce one bounded Codex invocation:

```text
work packet
-> role profile
-> context package
-> rendered prompt
-> codex exec
-> event log
-> last message return
-> tests
-> proof gate
-> receipt
```

Default local command shape:

```bash
mkdir -p ION/05_context/current/codex_cli

codex exec --sandbox workspace-write \
  --output-last-message ION/05_context/current/codex_cli/latest_return.md \
  < ION/05_context/current/codex_cli/latest_prompt.md
```

Optional event capture:

```bash
codex exec --json --sandbox workspace-write \
  --output-last-message ION/05_context/current/codex_cli/latest_return.md \
  < ION/05_context/current/codex_cli/latest_prompt.md \
  > ION/05_context/current/codex_cli/latest_events.jsonl
```

### Lane D — Return intake

Every Codex return must include:

```text
### CONTEXT PROOF
### TEMPLATE ACTION PROOF
### RESULT
```

Raw Codex output remains proposal until proof gates and Steward integration accept it.
